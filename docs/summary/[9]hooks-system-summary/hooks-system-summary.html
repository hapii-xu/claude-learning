一、概念与整体架构

Hook 是在 Claude Code 执行过程的关键节点注入自定义逻辑的机制 —— 在工具调用前/后、会话生命周期、权限决策等时机触发用户定义的 Shell 命令、LLM 子调用、HTTP 请求或 Agent 子代理。

在哪里配置

配置文件范围Git 追踪
~/.claude/settings.json全局（所有项目）否
.claude/settings.json项目级（团队共享）是
.claude/settings.local.json项目本地（不提交）否
Skill 的 SKILL.md frontmatter该 Skill 激活时视 Skill 位置

基础配置结构

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash(git *)",
        "hooks": [
          { "type": "command", "command": "echo \"将要执行 git 命令\"" }
        ]
      }
    ],
    "PostToolUse": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "logger -t claude \"工具调用完成\"",
            "async": true
          }
        ]
      }
    ]
  }
}

二、4 种 Hook 类型完整字段

1
command（Shell 命令）最常用

通过 Shell 执行任意命令，stdin 接收 $ARGUMENTS JSON。

{
  "type": "command",
  "command": "python3 /home/user/hooks/check.py",
  "shell": "bash",
  "timeout": 30,
  "if": "Bash(git push*)",
  "statusMessage": "安全检查中...",
  "once": false,
  "async": false,
  "asyncRewake": false
}
字段类型默认说明
type"command"—固定值
commandstring必填要执行的 Shell 命令，可用 $ARGUMENTS 读输入
shell"bash" \| "powershell""bash"Shell 解释器类型
timeoutnumber60超时秒数
ifstring—二级过滤条件，与 matcher 相同的权限规则格式
statusMessagestring—执行时 spinner 显示的文字
oncebooleanfalsetrue = 执行一次后从内存中移除（不改文件）
asyncbooleanfalsetrue = 后台执行不阻塞，模型立即继续
asyncRewakebooleanfalsetrue = 后台执行，exit 2 时唤起模型（隐含 async）

读取 $ARGUMENTS 的方法

# 方法 1：Python（推荐，处理 JSON 最可靠）
echo $ARGUMENTS | python3 -c "
import json, sys
data = json.load(sys.stdin)
print(data.get('tool_name'))
print(data.get('tool_input', {}).get('command'))
"

# 方法 2：jq（轻量级）
TOOL=$(echo $ARGUMENTS | jq -r '.tool_name')
CMD=$(echo $ARGUMENTS | jq -r '.tool_input.command // empty')

# 方法 3：Node.js
echo $ARGUMENTS | node -e "const d=JSON.parse(require('fs').readFileSync('/dev/stdin','utf8'));console.log(d.tool_name)"

# 方法 4：直接透传给脚本
python3 /path/to/my_hook.py <<< "$ARGUMENTS"

返回结构化 JSON

# 通过 stdout 输出 JSON 可精细控制行为
echo '{"continue": false, "stopReason": "检测到危险操作，已阻止"}'

# 或直接 exit 2 让模型看到 stderr
echo "禁止执行此操作！" >&2
exit 2
2
prompt（LLM 子调用）需要语义理解

让 LLM（默认 Haiku）来评估 Hook 输入并做决策。适合需要语义理解的场景。

{
  "type": "prompt",
  "prompt": "以下工具调用即将执行：$ARGUMENTS\n请判断是否存在安全风险...",
  "model": "claude-haiku-4-5-20251001",
  "timeout": 30,
  "if": "Bash(curl*)",
  "statusMessage": "AI 安全审查中..."
}
字段类型默认说明
type"prompt"—固定值
promptstring必填给 LLM 的提示词，支持 $ARGUMENTS 占位符
modelstring快速小模型如 claude-haiku-4-5-20251001
timeoutnumber30超时秒数
ifstring—二级过滤
3
http（HTTP POST）外部服务集成

向外部服务发送 Hook 输入，适合通知、监控、审计系统集成。

{
  "type": "http",
  "url": "https://your-server.com/hooks/claude-code",
  "headers": {
    "Content-Type": "application/json",
    "Authorization": "Bearer $MY_API_TOKEN",
    "X-Project": "${PROJECT_NAME}"
  },
  "allowedEnvVars": ["MY_API_TOKEN", "PROJECT_NAME"],
  "timeout": 10,
  "if": "Bash(git push*)"
}
字段类型默认说明
type"http"—固定值
urlstring必填POST 目标 URL，必须 https（本地除外）
headersRecord{}额外请求头，值中可用 $VAR 或 ${VAR} 引用环境变量
allowedEnvVarsstring[][]必须显式声明才能在 headers 中使用的环境变量名
timeoutnumber30超时秒数
⚠️ SSRF 防护：http Hook 的 URL 会经过内网地址检测（127.0.0.1、10.x.x.x、192.168.x.x 等），访问内网 IP 会被阻断。
4
agent（Agent 子代理）最强验证能力

启动一个独立子代理，可执行多步骤验证逻辑，比 prompt 类型功能更强（可调用工具）。

{
  "type": "agent",
  "prompt": "你是安全审查专家。以下操作即将执行：$ARGUMENTS\n请分析风险，调用 SyntheticOutput 返回 {ok, reason}",
  "model": "claude-haiku-4-5-20251001",
  "timeout": 60,
  "if": "Bash(rm -rf*)"
}
字段类型默认说明
type"agent"—固定值
promptstring必填给 Agent 的指令，支持 $ARGUMENTS
modelstringHaiku默认使用 Haiku 做快速验证
timeoutnumber60超时秒数

三、27 种 Hook 事件完整参考

按生命周期分组。每个事件给出触发时机、$ARGUMENTS 输入结构、exit code 行为、hookSpecificOutput 输出。

3.1 工具生命周期

P
PreToolUse — 工具执行前

触发时机：模型决定调用某个工具之后、工具实际执行之前。

{
  "tool_name": "Bash",
  "tool_input": { "command": "npm test" },
  "tool_use_id": "toolu_abc123"
}
exit code效果
0成功，stdout/stderr 在 transcript 模式可见
2阻断工具执行，stderr 显示给模型（模型收到错误消息）
其他显示 stderr 给用户（不影响工具继续执行）

hookSpecificOutput 支持

{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow",
    "permissionDecisionReason": "已通过自动化审查",
    "updatedInput": { "command": "npm test -- --coverage" },
    "additionalContext": "注意：此命令可能耗时较长"
  }
}
字段说明
permissionDecision"allow" / "deny" / "ask" — 覆盖权限系统的决策
permissionDecisionReason展示给用户的权限决策原因
updatedInput替换工具输入（如修改命令内容）
additionalContext注入到下一轮对话的补充上下文（给模型看）
P
PostToolUse — 工具执行成功后
 
{
  "tool_name": "Bash",
  "tool_input": { "command": "npm test" },
  "tool_use_id": "toolu_abc123",
  "tool_response": { "output": "All tests passed!", "exit_code": 0 }
}
exit code效果
0成功，stdout 在 transcript 模式显示
2将 stderr 作为新消息显示给模型（模型继续对话）
其他显示 stderr 给用户（不影响对话）
 
{
  "hookSpecificOutput": {
    "hookEventName": "PostToolUse",
    "additionalContext": "测试运行完成，覆盖率: 87%",
    "updatedMCPToolOutput": { "enhanced": "result" }
  }
}
P
PostToolUseFailure — 工具执行失败后
 
{
  "tool_name": "Bash",
  "tool_input": { "command": "npm test" },
  "tool_use_id": "toolu_abc123",
  "error": "Process exited with code 1",
  "error_type": "execution_error",
  "is_interrupt": false,
  "is_timeout": false
}

exit code 行为与 PostToolUse 相同：0 显示 stdout，2 立即显示 stderr 给模型，其他显示给用户。

3.2 权限生命周期

P
PermissionRequest — 权限弹框弹出时
 
{
  "tool_name": "Bash",
  "tool_input": { "command": "rm -rf dist/" },
  "tool_use_id": "toolu_abc123"
}

自动 allow

{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "allow",
      "updatedInput": { "command": "rm -rf dist/ build/" }
    }
  }
}

自动 deny

{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "deny",
      "message": "不允许删除 dist 目录",
      "interrupt": true
    }
  }
}
P
PermissionDenied — 权限被拒绝后
 
{
  "tool_name": "Bash",
  "tool_input": { "command": "sudo rm -rf /" },
  "tool_use_id": "toolu_abc123",
  "reason": "Dangerous system command"
}
{
  "hookSpecificOutput": {
    "hookEventName": "PermissionDenied",
    "retry": true   // 让模型知道可以重试（以不同方式）
  }
}

3.3 会话生命周期

S
SessionStart — 会话启动
 
{
  "source": "startup",   // startup | resume | clear | compact
  "session_id": "sess_abc123"
}
exit code效果
0stdout 作为上下文发送给 Claude
其他stderr 显示给用户，不影响启动
 
{
  "hookSpecificOutput": {
    "hookEventName": "SessionStart",
    "additionalContext": "当前项目：5 个未解决 PR，覆盖率 78%",
    "initialUserMessage": "请先检查 TODO 列表",
    "watchPaths": ["/path/to/watch1", "/path/to/watch2"]
  }
}
S
Setup · SessionEnd · Stop · StopFailure

Setup（仓库初始化）

{ "trigger": "init" }   // init | maintenance

SessionEnd（会话结束）

{ "reason": "prompt_input_exit" }   // clear | logout | prompt_input_exit | other

Stop（响应结束前）— exit code 2 可强制模型继续

{ "session_id": "sess_abc123" }
exit code效果
0stdout/stderr 不显示
2将 stderr 作为新消息显示给模型（模型被迫继续对话）
其他stderr 只显示给用户

StopFailure（API 错误中断，不支持阻断）

{
  "error": "rate_limit",   // rate_limit|authentication_failed|billing_error|invalid_request|server_error|max_output_tokens|unknown
  "error_message": "Too many requests"
}

3.4 子代理 / 压缩 / 用户输入

A
SubagentStart · SubagentStop

SubagentStart（子代理启动）

{ "agent_id": "agent_abc123", "agent_type": "claude" }

stdout 显示给子代理，不支持阻断。

SubagentStop（子代理结束）— exit code 2 让子代理继续工作

{
  "agent_id": "agent_abc123",
  "agent_type": "claude",
  "agent_transcript_path": "/tmp/transcripts/agent_abc123.json"
}
C
PreCompact · PostCompact

PreCompact（压缩前）

{
  "trigger": "auto",   // manual | auto
  "messages_count": 150,
  "estimated_tokens": 180000
}
exit code效果
0stdout 作为自定义压缩指令追加
2阻断压缩
其他stderr 显示给用户，继续压缩

PostCompact（压缩后）

{
  "trigger": "auto",
  "summary": "对话摘要内容..."
}
U
UserPromptSubmit — 用户提交消息时
 
{
  "prompt": "请帮我审查这段代码",
  "session_id": "sess_abc123"
}
exit code效果
0stdout 作为额外上下文发送给 Claude
2阻断处理，清除原始提示词，stderr 只显示给用户
其他stderr 显示给用户，不影响处理
 
{
  "hookSpecificOutput": {
    "hookEventName": "UserPromptSubmit",
    "additionalContext": "当前 git branch: feature/auth"
  }
}

3.5 文件/目录/通知/MCP

F
CwdChanged · FileChanged

CwdChanged（工作目录变更）

{
  "old_cwd": "/home/user/project-a",
  "new_cwd": "/home/user/project-b"
}
特殊环境变量：$CLAUDE_ENV_FILE — 向此文件写入 export VAR=value 格式内容，可为后续 BashTool 命令设置环境变量。
{
  "hookSpecificOutput": {
    "hookEventName": "CwdChanged",
    "watchPaths": ["/home/user/project-b/.env"]
  }
}

FileChanged（监听的文件变化）

matcher 字段指定要监听的文件名模式（如 ".envrc|.env"），用 | 分隔。

{
  "file_path": "/home/user/project/.env",
  "event": "change"   // change | add | unlink
}
N
Notification — 发送通知时
 
{
  "notification_type": "permission_prompt",
  "message": "Claude 请求执行 Bash 命令"
}

notification_type 可选值：

  • permission_prompt — 权限确认弹框
  • idle_prompt — 用户长时间未操作
  • auth_success — 认证成功
  • elicitation_dialog / elicitation_complete / elicitation_response — MCP 触发的弹框
E
Elicitation · ElicitationResult

Elicitation（MCP 请求用户输入时）

{
  "mcp_server_name": "my-mcp-server",
  "message": "请输入数据库连接密码",
  "requested_schema": { "type": "object", "properties": { "password": { "type": "string" } } },
  "elicitation_id": "elicit_abc123"
}
{
  "hookSpecificOutput": {
    "hookEventName": "Elicitation",
    "action": "accept",   // accept | decline | cancel
    "content": { "password": "my-secret-password" }
  }
}

ElicitationResult（用户响应后）

{
  "mcp_server_name": "my-mcp-server",
  "action": "accept",
  "content": { "password": "***" },
  "mode": "user",
  "elicitation_id": "elicit_abc123"
}

3.6 配置 / Worktree / Coordinator

W
WorktreeCreate · WorktreeRemove — 功能型 Hook

WorktreeCreate 是特殊功能型 Hook：stdout 应该输出创建的 worktree 绝对路径。

WorktreeCreate 输入

{ "name": "suggested-worktree-slug" }

示例 command（用 git worktree 实现）

#!/bin/bash
NAME=$(echo $ARGUMENTS | python3 -c "import json,sys; print(json.load(sys.stdin)['name'])")
WORKTREE_PATH="/tmp/worktrees/$NAME"
git worktree add "$WORKTREE_PATH" HEAD
echo "$WORKTREE_PATH"   # stdout 必须输出绝对路径

WorktreeRemove 输入

{ "worktree_path": "/tmp/worktrees/suggested-worktree-slug" }
C
ConfigChange · InstructionsLoaded

ConfigChange（配置文件变更）— exit code 2 可阻断变更

{
  "source": "project_settings",   // user_settings|project_settings|local_settings|policy_settings|skills
  "file_path": "/project/.claude/settings.json"
}

InstructionsLoaded（指令文件加载，纯观测不可阻断）

{
  "file_path": "/project/.claude/CLAUDE.md",
  "memory_type": "Project",   // User | Project | Local | Managed
  "load_reason": "session_start",   // session_start|nested_traversal|path_glob_match|include|compact
  "globs": ["src/**/*.ts"],
  "trigger_file_path": "/project/src/main.ts"
}
T
TeammateIdle · TaskCreated · TaskCompleted

Coordinator 多代理模式（这三类事件 exit code 2 可阻止对应行为）。

TeammateIdle（队友即将空闲）

{ "teammate_name": "agent-1", "team_name": "dev-team" }

TaskCreated / TaskCompleted（任务创建/完成）

{
  "task_id": "task_abc123",
  "task_subject": "实现用户登录功能",
  "task_description": "需要包含 OAuth2 支持...",
  "teammate_name": "agent-1",
  "team_name": "dev-team"
}

四、通用输出字段

所有 Hook 类型的 stdout JSON 都支持以下顶层字段。

{
  "continue": true,
  "suppressOutput": false,
  "stopReason": "拒绝原因（continue=false 时展示）",
  "decision": "approve",
  "reason": "原因说明",
  "systemMessage": "注入到下一轮 system prompt 的文字"
}
字段类型说明
continuebooleanfalse = 中止整个对话（等效用户按 Esc），true = 继续（默认）
suppressOutputbooleantrue = 隐藏此 Hook 的 stdout，不显示在 transcript 模式
stopReasonstringcontinue: false 时显示给用户的说明
decision"approve" \| "block"用于展示决策（不直接控制流程）
reasonstring决策原因说明
systemMessagestring注入到下一轮 system message 的内容

五、matcher 字段详解

matcher 在 HookMatcher 层级定义，决定整组 Hook 何时匹配。格式：ToolName(pattern) 或省略（匹配所有）。

{
  "PreToolUse": [
    { "matcher": "Bash(git *)", "hooks": [...] },
    { "matcher": "FileWriteTool(src/**/*.ts)", "hooks": [...] },
    { "hooks": [...] }   // 省略 matcher = 匹配所有工具
  ]
}
matcher 示例匹配范围
省略所有工具
"Bash(*)"所有 Bash 调用
"Bash(git *)"git 开头的 Bash 命令
"Bash(npm install)"精确匹配 npm install
"FileWriteTool(src/**)"写入 src 目录
"Read(*.ts)"读取 .ts 文件
if 字段：Hook 级别的第二道过滤，格式与 matcher 相同。即使外层 matcher 是 Bash(*)，加 "if": "Bash(git push*)" 后只有 push 命令会触发此 hook。

在 SKILL.md 中内联定义 Hooks

---
description: 带完整检查流程的安全提交
argument-hint: "<commit message>"
allowed-tools: [BashTool, Read]
hooks:
  PreToolUse:
    - matcher: "Bash(git commit*)"
      hooks:
        - type: command
          command: |
            STAGED=$(git diff --cached --name-only 2>/dev/null)
            if [ -z "$STAGED" ]; then echo "❌ 无暂存文件" >&2; exit 2; fi
            echo "✅ 暂存：$STAGED"
          statusMessage: 验证暂存文件...
  PostToolUse:
    - matcher: "Bash(git commit*)"
      hooks:
        - type: command
          command: echo "✅ 提交完成：$(git log --oneline -1)"
          async: true
---

# 安全提交技能
请按以下步骤：1. git status 2. git diff --cached 3. git commit -m "$ARGUMENTS"

七、Hook 执行时序与多 Hook 合并

同一事件多 Hooks 的执行顺序

1
Matcher 1 (Bash(git *)) → Hook A → Hook B
结果 A, 结果 B
2
Matcher 2 (Bash(*)) → Hook C
结果 C
3
Matcher 3 (无 matcher) → Hook D
结果 D
  • 多个 Hook 串行执行（一个结束后才执行下一个）
  • 任何一个 Hook 返回 exit code 2 或 continue: false，即阻断工具执行
  • 多个 additionalContext 按顺序合并

1

2

十、快速参考卡

哪个 Hook 该用什么 exit code？

PreToolUse

  • 0 记录/不干预
  • 2 阻断工具执行（模型收到错误）
  • 其他 记录错误（不影响执行）

PostToolUse

  • 0 记录/不干预
  • 2 给模型发送追加消息（模型继续）
  • 其他 记录错误（不影响对话）

Stop

  • 0 不干预
  • 2 强制模型继续工作
  • 其他 记录错误

UserPromptSubmit

  • 0 stdout 作为额外上下文
  • 2 阻断此次提问
  • 其他 记录错误

SessionStart

  • 0 stdout 作为上下文发给 Claude
  • 其他 记录错误（不影响启动）

PreCompact

  • 0 stdout 作为压缩指令
  • 2 阻断压缩
  • 其他 继续压缩
