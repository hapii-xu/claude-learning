# 终端 UI 层知识总结

> 这是"总结学习"栏目的第十二篇。目标：理解 Claude Code 的**终端界面是如何构建的**——从 `packages/@ant/ink` 这个 forked Ink 框架的底层能力，到 `REPL.tsx` 如何把流式 `AssistantMessage` 变成可交互的终端 UI，再到 `Messages.tsx` 里针对长会话的六重渲染优化。

## 一、终端 UI 层全貌

### 层次结构

```
src/screens/REPL.tsx                     ← 主屏，6687 行
│   ├── src/components/Messages.tsx      ← 对话消息渲染，1114 行
│   │   └── MessageRow.tsx              ← 单条消息渲染
│   ├── src/components/PromptInput/      ← 用户输入区域
│   │   └── PromptInput.tsx             ← 输入主组件，1000+ 行
│   ├── src/components/permissions/     ← 工具权限弹框
│   │   └── PermissionRequest.tsx       ← 按工具类型路由
│   └── ... (149 个组件)
│
packages/@ant/ink/                       ← Forked Ink 框架
    core/        ← 渲染协调器、layout 引擎、terminal I/O
    components/  ← Box, Text, ScrollBox, App 等原语
    hooks/       ← useInput, useTerminalSize, useAnimation...
    theme/       ← 主题系统 + Dialog/Tabs/ProgressBar 等设计组件
    keybindings/ ← 和弦式快捷键系统
```

### 为什么 fork Ink

官方 Ink 是轻量级终端 React 渲染器，但 Claude Code 需要：
- **大型对话的高性能渲染**：虚拟化长消息列表
- **复杂快捷键系统**：和弦（chord）组合键支持
- **高级事件系统**：鼠标事件、终端焦点、OSC 协议通知
- **优化的 Screen 缓冲**：`seenDirtyChild` 级联，只写脏节点，避免长会话的无效重绘

### 1. `packages/@ant/ink` 框架层次

**145 个 TypeScript 文件**，分 6 层：

| 层 | 目录 | 内容 |
|---|------|------|
| core | `core/` 40+ 文件 | 渲染协调器（reconciler）、Yoga layout 引擎、terminal I/O、Screen buffer |

**关键优化**：`renderChildren` 中的 `seenDirtyChild` 级联——只有有脏子节点时才重写对应的 Screen buffer 区域，长会话中大多数消息行不重绘。

### 2. REPL.tsx 主屏架构

`src/screens/REPL.tsx` 是整个交互界面的 root，6687 行，核心职责：

| 职责 | 关键位置 | 说明 |
|------|---------|------|

**后台会话流程**（Ctrl+B，line 3057-3147）：
```
Ctrl+B 触发
    ↓

    ↓
startBackgroundSession({               ← 用当前状态启动后台任务
  messages,
  queryParams,
  description,        ← 终端标题
  agentDefinition
})
    ↓
前台 REPL 恢复可输入状态
后台任务在 Background Task Indicator 中显示进度
```

### 3. Messages.tsx 的六重渲染优化

`src/components/Messages.tsx`（1114 行）是性能最敏感的组件，针对"几百轮对话"场景做了六重优化：

**① 昂贵变换与廉价切片分离**（line 536-686）：
```
昂贵变换 memo（536-668）：过滤、重排、分组、折叠、反查 Map 构建
                          → 只在 messages 真正变化时重算（非滚动时）
廉价切片（671-686）：         非虚拟化路径的窗口切片
                          → 每次渲染都执行，但极轻量
```
分离原因：滚动不改变消息内容，不应触发 O(n) 的 Map 重建。

**② expandKey 工具与结果分组**（line 985-989）：
```typescript
// tool_use 和 tool_result 用同一 key 分组 → 渲染在同一区块内
expandKey = (msg) => msg.tool_use_id || msg.uuid
```

**③ 自定义 memo 比较器**（line 991-1052）：
```
streamingToolUses: 检查 contentBlock 内容（引用变了但内容没变时 skip）
inProgressToolUseIDs: setsEqual() 集合比较
unseenDivider: uuid + count 两字段比较
tools: 比较 tool name 列表（不比较整个 Tool 对象）
```

**④ `shouldRenderStatically()`**（line 1054-1114）：
已完成的消息（非流式、非 in-progress）标记为静态，不参与增量更新，避免历史消息的无效重渲染。

**⑤ 虚拟化（VirtualMessageList）**（line 928-952）：
当 `scrollRef` 存在（全屏模式）时启用，只渲染视窗内的消息行。Ctrl+E 可控制截断上限（默认 30 条）。

**⑥ 流式 thinking 超时**（line 407-420）：
```
streaming thinking 块显示 30s 后自动隐藏（防止 stuck thinking 占满屏幕）
threshold: 30_000ms after stream ends
```

### 5. PermissionRequest.tsx 工具路由

### 6. Keybinding 系统

`packages/@ant/ink/keybindings/` 实现**和弦式（chord）快捷键**：

```
用户按键序列输入
    ↓
keybindingParser：解析为 KeyChord
    ↓
keybindingMatcher：与注册的绑定匹配（支持前缀等待）
    ↓
keybindingResolver：调用匹配的 handler

示例：Ctrl+X → Ctrl+C  可以作为一个二段和弦绑定
```

`GlobalKeybindingHandlers` + `CommandKeybindingHandlers` 在 REPL.tsx 中注册（line 159-162），覆盖 Ctrl+C（取消）、Ctrl+R（历史搜索）、Ctrl+B（后台）、Ctrl+E（截断控制）等。

## 三、应该了解的 3 个重要机制

### A. LogoHeader 的 OffscreenFreeze

`Messages.tsx` line 83-101：

```tsx
<OffscreenFreeze freeze={isScrolledBack}>
  <LogoHeader agentDefinitions={agentDefinitions} />
</OffscreenFreeze>
```

当用户向上滚动查看历史消息时（`isScrolledBack=true`），Logo header 被"冻结"（停止渲染更新），避免频繁的 Agent 状态变化触发顶部区域重渲染。

### B. Brief 模式过滤

`Messages.tsx` 的 `filterForBriefTool()`（line 127-192）和 `dropTextInBriefTurns()`（line 203-237）：

在 Brief Tool 激活时，只显示 Brief tool 块 + 工具结果 + 真实用户输入，过滤掉大量助手文本，让界面更简洁。

### C. 终端能力检测

`packages/@ant/ink/utils/` 中检测当前终端的能力：
- 超链接支持（OSC 8）
- 鼠标跟踪支持
- Xterm.js 检测（VS Code 内嵌终端）
- ANSI sync output（`\x1b[?2026h` 减少闪烁）
- `TerminalNotification` hook：OSC 9;4 进度上报（Windows Terminal 进度条）

## 五、学习建议

1. **从 REPL.tsx 顶部的 import 出发**：列出所有导入的组件和 hook，构建依赖图——这能快速建立整个 UI 层的知识框架
