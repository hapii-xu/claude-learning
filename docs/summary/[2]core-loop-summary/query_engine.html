四、构造函数与延迟启动

constructor(config: QueryEngineConfig) {   // src/QueryEngine.ts:238
  this.config = config
  this.mutableMessages = config.initialMessages ?? []
  this.readFileState = config.readFileCache
  this.abortController = config.abortController ?? createAbortController()
  this.permissionDenials = []
  this.totalUsage = EMPTY_USAGE
}

构造函数仅做赋值——没有文件 I/O，没有网络请求，没有 React 上下文访问。

所有重量级操作（提示词 fetch、skill 加载、API 调用）全部推迟到第一次 submitMessage() 调用时执行。这种设计使得 QueryEngine 的实例化成本接近于零。

8.2 三阶段累计（:871-922）

case 'stream_event': {
  const { event } = message

  // 阶段 1：message_start — 用 API 返回的 input tokens 初始化
  if (event.type === 'message_start') {
    currentMessageUsage = EMPTY_USAGE
    currentMessageUsage = updateUsage(currentMessageUsage, event.message.usage)
  }

  // 阶段 2：message_delta — 累积 output tokens（stop_reason 也在这里）
  if (event.type === 'message_delta') {
    currentMessageUsage = updateUsage(currentMessageUsage, event.usage)
    lastStopReason = event.delta.stop_reason  // 真实 stop_reason 在此获取
  }

  // 阶段 3：message_stop — 提交到实例级 totalUsage
  if (event.type === 'message_stop') {
    this.totalUsage = accumulateUsage(this.totalUsage, currentMessageUsage)
    try { _usageSink?.(this.totalUsage, mainLoopModel) } catch {}
  }
}

8.3 updateUsage 的"非零字段优先"语义

Anthropic API 有时在 message_delta 里对稳定字段（如 cache_creation_input_tokens）发送显式 0，而真实值已在 message_start 里发过了。updateUsage 采用"非零值优先"策略：只在新值 > 0 时才覆盖旧值，防止 0 覆盖正确的缓存 token 数。

9.3 为什么有这种分裂

  • 历史原因：REPL 是最早的代码，query() 是为 REPL 设计的
  • 设计原因：QueryEngine 是后来为 SDK / ACP 场景封装的稳定 session API，为避免打破 REPL 的微妙状态机，没有让 REPL 迁移到 QueryEngine
  • 协议原因：SDKMessage 协议包含会计信息（cost、duration、permission_denials）——这些是程序消费者需要的，UI 并不关心

11.3 error_during_execution 的 ede_diagnostic 排错串

当 isResultSuccessful 失败时，errors 数组包含：

12.1 自动 compact 边界的两次清理

当 query() 发出 system/compact_boundary 时，QueryEngine 做两次不同目的的处理：

第一次：持久化前裁剪（:773-789）——在 yield 之前，将 boundary 前的消息刷写到磁盘，确保 --resume 时能找到正确的 tailUuid：

const tailUuid = message.compactMetadata?.preservedSegment?.tailUuid
if (tailUuid) {
  const idx = this.mutableMessages.findLastIndex(m => m.uuid === tailUuid)
  if (idx !== -1) await recordTranscript(this.mutableMessages.slice(0, idx + 1))
}

第二次：内存裁剪（:1019-1043）——boundary 消息 push 后，删除其之前的所有消息，防止长会话内存无限增长：

const idx = this.mutableMessages.length - 1
if (idx > 0) this.mutableMessages.splice(0, idx)  // 只保留 boundary 消息本身

12.2 HISTORY_SNIP 的 snipReplay 钩子

每次 assistant 消息 push 后触发（:1009-1016）：

const snipResult = this.config.snipReplay?.(msg, this.mutableMessages)
if (snipResult?.executed) {
  this.mutableMessages.length = 0
  this.mutableMessages.push(...snipResult.messages)
}
break  // 终止本 turn 的事件消费

整体替换 mutableMessages，清除 tombstone 标记，防止长会话中 tombstone 累积导致内存泄漏。

12.3 文件历史快照（fire-and-forget）

// :708-723
messagesFromUserInput.forEach(message => {
  void fileHistoryMakeSnapshot(updaterCallback, message.uuid)  // fire-and-forget

在用户消息 UUID 下备份所有被工具读写的文件，支持 /undo 功能。用 void 丢弃 Promise，不阻塞主流程。

12.4 归因仅透传

QueryEngine 把 updateAttributionState 回调注入到工具执行 context 中，由文件编辑工具在修改文件时调用，记录 git commit 归因数据。QueryEngine 本身从不读取 appState.attribution。

十三、ask() 包装函数

定义：src/QueryEngine.ts:1328-1441。供 src/cli/print.ts 使用的一次性包装器。

readFileCache 克隆（:1401）
cloneFileStateCache(getReadFileCache())，防止临时 engine 实例污染调用方的文件缓存状态。
finally 回灌（:1439）

export async function* ask({ prompt, cwd, ... }) {
  const engine = new QueryEngine({
    ...
    readFileCache: cloneFileStateCache(getReadFileCache()),  // 克隆
    ...
  })
  try {
    yield* engine.submitMessage(prompt, { uuid, isMeta })
  } finally {
    setReadFileCache(engine.getReadFileState())  // 回灌
  }
}

14.3 Eager flush

在每个 result 消息 yield 后（包括错误路径），当 CLAUDE_CODE_EAGER_FLUSH 或 CLAUDE_CODE_IS_COWORK 环境变量存在时，执行 await flushSessionStorage()。

原因：某些宿主进程（如 claude-desktop）在收到 result 消息后立即杀死 CLI 进程，eager flush 确保会话存储在此之前全部落盘。

14.4 Transcript 写盘策略

消息类型写法原因
assistant void recordTranscript()（fire-and-forget） claude.ts 会在 message_delta 时回填 message.usage，await 会输给 100ms drain timer，导致写入丢 usage 字段
其他所有类型 await recordTranscript() enqueueWrite 保序，安全可 await
