三、⭐ wrappedCanUseTool：权限拒绝追踪

349-377 把外部传入的 canUseTool 包一层，在放行/拒绝之外记账拒绝：

const wrappedCanUseTool: CanUseToolFn = async (tool, input, ctx, msg, toolUseID, force) => {
  const result = await canUseTool(tool, input, ctx, msg, toolUseID, force)
  if (result.behavior !== 'allow') {
    this.permissionDenials.push({
      type: 'permission_denial',
      tool_name: sdkCompatToolName(tool.name),
      tool_use_id: toolUseID,
      tool_input: input,
    })
  }
  return result
}
  • 透明代理：原样转发参数与返回值，只在 behavior !== 'allow' 时副作用式 push 一条拒绝记录。
  • sdkCompatToolName(tool.name)：把内部工具名转成 SDK 兼容名（如 MCP 工具的命名）。
  • 这条 permissionDenials 数组最终出现在每个 result 消息的 permission_denials 字段里（[10]），让 SDK 调用方知道哪些工具被拒。
为什么不在 canUseTool 内部记：canUseTool 是外部注入的（REPL/SDK 各有实现），engine 不该要求每个实现都自带记账。包一层是关注点分离：权限判定归 canUseTool，拒绝统计归 engine。

四、初始模型与 thinking 配置

决策优先级
模型userSpecifiedModel（显式指定）> getMainLoopModel()（全局默认）
thinking显式 thinkingConfig > shouldEnableThinkingByDefault() → adaptive / disabled
  • initialAppState 快照一次（后续 fastMode、toolPermissionContext 都从它取，避免重复 getAppState()）。
  • initialMainLoopModel 是「初始」值——[6] 里 slash 命令可能通过 modelFromUserInput 覆盖它，得到最终 mainLoopModel。
adaptive vs disabled：adaptive 让模型按需自行决定思考预算；disabled 完全关闭 thinking。默认开（除非环境/配置显式关）。
