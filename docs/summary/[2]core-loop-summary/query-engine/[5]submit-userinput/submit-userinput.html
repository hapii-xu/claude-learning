一、processUserInputContext：第一次构建（会修改 messages）

processUserInputContext 传给 processUserInput（src/utils/processUserInput/index.ts），后者用这个上下文来执行 slash 命令、处理输入、可能修改消息列表。

字段作用
messages + setMessages会修改消息数组的 slash 命令通过 setMessages 回写；print 模式写回 mutableMessages，交互模式写 AppState
handleElicitation处理 MCP -32042 错误触发的 URL 询问
updateFileHistoryState / updateAttributionState让 slash 命令能更新 fileHistory / attribution 状态
setMessages 在交互模式下会通过 setAppState 写 AppState.messages；在 print 模式下直接写 this.mutableMessages——让后续的 push和 fileHistory 快照能看到变化。

二、handleOrphanedPermission：孤立权限处理（每 engine 一次）

if (orphanedPermission && !this.hasHandledOrphanedPermission) {
  this.hasHandledOrphanedPermission = true
  for await (const message of handleOrphanedPermission(
    orphanedPermission,
    tools,
    this.mutableMessages,
    processUserInputContext,
  )) {
    yield message
  }
}
  • 孤立权限：前一轮中，用户对某个工具权限请求进行了操作（如 allow），但随后对话被中断/退出，权限结果没来得及写回状态。本轮启动时，orphanedPermission 携带这个「孤儿」请求。
  • 每 engine 一次：hasHandledOrphanedPermission 守卫确保只处理一次——避免重复 yield。
  • handleOrphanedPermission 的产出（消息）立即 yield，不进 mutableMessages。

三、⭐ processUserInput：分发输入

processUserInput 是分发中枢：

普通文本 promptshouldQuery: true + messagesFromUserInput（转成 user 消息）
特殊命令（如 /model 切模型）modelFromUserInput 覆盖初始模型
  • allowedTools：slash 命令可能更新「始终允许的工具列表」。
  • resultText：本地命令的结果文本（!shouldQuery 短路路径会用，见 [7]）。
querySource: 'sdk'：标记这是 headless/SDK 路径的调用（vs REPL 的 'repl'），影响权限提示等行为。

四、push 新消息 + 快照

this.mutableMessages.push(...messagesFromUserInput)
const messages = [...this.mutableMessages]
  • push 到 mutableMessages：这是本 engine 的真理之源（会话状态）。
  • 快照副本：messages = [...this.mutableMessages]，后续传给 query()（[8]）——query 循环内 push 的消息会追加到这个副本，最终 submitMessage 结束时要把副本的增量同步回 mutableMessages。
为何快照：query() 内部会修改消息数组。如果直接传 mutableMessages，query 内的副作用会直接影响 engine 状态；快照后，engine 可以在 query 结束时选择性同步。
