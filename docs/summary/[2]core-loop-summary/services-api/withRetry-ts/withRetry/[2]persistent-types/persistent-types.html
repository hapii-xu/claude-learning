一、Persistent（无人值守）重试配置

1.1 三个常量

const PERSISTENT_MAX_BACKOFF_MS = 5 * 60 * 1000        // 5 分钟：退避上限
const PERSISTENT_RESET_CAP_MS = 6 * 60 * 60 * 1000     // 6 小时：窗口 reset 等待上限
const HEARTBEAT_INTERVAL_MS = 30_000                   // 30 秒：心跳分块间隔
常量值作用
PERSISTENT_MAX_BACKOFF_MS5 分钟persistent 模式指数退避的封顶（普通模式封 32 秒）
PERSISTENT_RESET_CAP_MS6 小时窗口限额 reset 等待 / 病态 retry-after 头的硬上限
HEARTBEAT_INTERVAL_MS30 秒长睡眠切块的粒度，每块 yield 一次心跳防 idle
// CLAUDE_CODE_UNATTENDED_RETRY：用于无人值守 session（仅 ant）。无限重试
// 429/529，使用更长的退避并周期性 yield keep-alive，避免宿主环境在等待中
// 把 session 标记为 idle。
// TODO(ANT-344)：通过 SystemAPIErrorMessage yield 做 keep-alive 只是临时方案，
// 直到有专门的 keep-alive 通道为止。
为什么要"心跳"：无人值守 session 在长等待（可能数小时）里若毫无 stdout 活动，宿主环境会判定它 idle 并回收。临时方案是把等待切成 30 秒块，每块 yield 一条 SystemAPIErrorMessage 制造 stdout 活动。这是借用错误消息通道做 keep-alive，故有 TODO(ANT-344) 标记将来要换专用通道。

1.2 门控：isPersistentRetryEnabled

function isPersistentRetryEnabled(): boolean {
  return feature('UNATTENDED_RETRY')
    ? isEnvTruthy(process.env.CLAUDE_CODE_UNATTENDED_RETRY)
    : false
}

双重门控：feature flag UNATTENDED_RETRY 且 环境变量 CLAUDE_CODE_UNATTENDED_RETRY 为真。feature 未开时直接 false，让相关分支能被外部构建 tree-shake。

1.3 两个错误分类器

function isTransientCapacityError(error: unknown): boolean {
  return (
    is529Error(error) || (error instanceof APIError && error.status === 429)
  )
}

function isStaleConnectionError(error: unknown): boolean {
  if (!(error instanceof APIConnectionError)) return false
  const details = extractConnectionErrorDetails(error)
  return details?.code === 'ECONNRESET' || details?.code === 'EPIPE'
}
函数判定用途
isTransientCapacityError529 或 429persistent 模式的"瞬时容量错误"——这类才无限重试（[7][10]）
isStaleConnectionErrorAPIConnectionError 且 ECONNRESET/EPIPE陈旧 keep-alive socket，重试前要禁连接池 + 换 client（[4]）

二、类型契约

2.1 RetryContext——降级状态的"就地传递"载体

export interface RetryContext {
  maxTokensOverride?: number     // max_tokens 溢出后调整的新上限
  model: string                  // 当前模型名
  thinkingConfig: ThinkingConfig // 思考预算
  fastMode?: boolean            // fast mode 是否仍激活
}

retryContext 在生成器里被创建一次，然后每次 operation(client, attempt, retryContext) 都把它传进去。关键在于它会被就地改写：

  • [5] 关闭 fast mode 时写 retryContext.fastMode = false。
  • [7] max_tokens 溢出时写 retryContext.maxTokensOverride = adjustedMaxTokens。

下一次 operation 就能看到这些降级，据此调整请求参数。

为什么用"就地改写副本"而非每次新建：降级是累积的——关了 fast mode、又调了 maxTokens，两个状态都要带到下一次尝试。一个可变的 retryContext 对象天然累积这些变更，比每次组装新对象简单。

2.2 RetryOptions——生成器的输入配置

interface RetryOptions {
  maxRetries?: number
  model: string
  fallbackModel?: string
  thinkingConfig: ThinkingConfig
  fastMode?: boolean
  signal?: AbortSignal
  querySource?: QuerySource
  initialConsecutive529Errors?: number   // ★ 预置 529 计数器
}

大部分字段顾名思义。重点是 initialConsecutive529Errors，源码注释解释了它的存在意义：

// 预置连续 529 计数器。当本重试循环是流式 529 之后的非流式 fallback 时
// 使用 —— 流式的 529 应计入 MAX_529_RETRIES，使触发 fallback 前的总 529
// 次数与命中过载的请求模式无关，保持一致。
场景：一次请求先以流式方式发，连续吃了 2 个 529，然后降级为非流式重发。非流式这一轮的 withRetry 如果从 0 开始数 529，就会比纯非流式路径多容忍几次过载。initialConsecutive529Errors=2 把流式已发生的 529 预置进计数器，让"触发模型 fallback 前总共容忍几次 529"这个语义与请求方式无关。

三、两个错误类

3.1 CannotRetryError——放弃重试

export class CannotRetryError extends Error {
  constructor(
    public readonly originalError: unknown,
    public readonly retryContext: RetryContext,
  ) {
    super(errorMessage(originalError))
    this.name = 'RetryError'
    if (originalError instanceof Error && originalError.stack) {
      this.stack = originalError.stack   // 保留原始 stack trace
    }
  }
}

携带两样东西上抛：originalError（真正的失败原因）+ retryContext（放弃时的降级状态快照）。保留原始 stack 让上层日志能定位真正的出错点，而不是 withRetry 内部的抛出点。

注意 this.name = 'RetryError'（不是 'CannotRetryError'）——历史命名遗留，上层按类型 instanceof 判断，不依赖 name 字符串。

3.2 FallbackTriggeredError——触发模型降级

export class FallbackTriggeredError extends Error {
  constructor(
    public readonly originalModel: string,
    public readonly fallbackModel: string,
  ) {
    super(`Model fallback triggered: ${originalModel} -> ${fallbackModel}`)
    this.name = 'FallbackTriggeredError'
  }
}
它不是"错误"，而是一个控制信号：连续 529 达阈值时抛出，携带 原模型 → fallback 模型，由 query.ts 捕获后切换模型重发整个请求（见 [0] 三类退出、[6]）。

四、速记口诀

  • Persistent 三常量：退避封顶 5min · reset 上限 6h · 心跳 30s（仅 ant + 双重门控）。
  • 心跳本质：借 SystemAPIErrorMessage 通道制造 stdout 活动防 idle（TODO 待换专用通道）。
  • RetryContext：可变副本，就地累积降级状态（fastMode / maxTokensOverride）传给下次 operation。
  • initialConsecutive529Errors：把流式已发生的 529 预置进非流式 fallback 的计数器，保持语义一致。
  • 两错误类：CannotRetryError（放弃，带原 stack）· FallbackTriggeredError（控制信号，切模型）。
