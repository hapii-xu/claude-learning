二、⭐ 前台 529 重试白名单

2.1 一个反直觉的决策：不是所有 529 都该重试

429（限流）几乎总该重试，但 529（服务过载）不同。529 意味着 Anthropic 后端正在容量级联——此时每一次重试都在给已经过载的网关加压。源码注释把这个放大效应说得很直白：

// 用户会阻塞等待结果的前台 query source —— 这些会在 529 时重试。
// 其他所有 source（摘要、标题、建议、分类器）都会立即放弃：在容量级联期间，
// 每次重试都会带来 3-10× 的网关放大，而用户根本看不到这些失败。新增的
// source 默认不重试 —— 仅当用户在等待结果时才加到这里。
关键判断标准：用户是否正阻塞等待这个结果？是 → 重试值得；否 → 放弃，省下的容量留给真正有人看的请求。

2.2 白名单成员

const FOREGROUND_529_RETRY_SOURCES = new Set<QuerySource>([
  'repl_main_thread',                          // REPL 主对话
  'repl_main_thread:outputStyle:custom',
  'repl_main_thread:outputStyle:Explanatory',
  'repl_main_thread:outputStyle:Learning',
  'sdk',                                        // SDK 调用
  'agent:custom', 'agent:default', 'agent:builtin',  // 子 agent
  'compact',                                    // 压缩（用户在等压缩完成）
  'hook_agent', 'hook_prompt',                  // hook 触发的查询
  'verification_agent',                         // 验证 agent
  'side_question',                              // 旁路提问
  'auto_mode',                                  // 安全分类器（auto-mode 正确性依赖它）
  ...(feature('BASH_CLASSIFIER') ? (['bash_classifier'] as const) : []),
])
类别成员为何前台
主对话repl_main_thread* / sdk用户直接在等回答
子 agentagent:custom/default/builtinagent 结果是用户任务的一部分
压缩 / 验证compact / verification_agent / side_question阻塞主流程，用户在等
Hookhook_agent / hook_prompthook 输出影响后续行为
安全分类器auto_mode / bash_classifier必须完成才能保证 auto-mode 正确性
几个易错点（源码注释特别澄清）：
  • 安全分类器 yoloClassifier.ts 用的是 'auto_mode'，不是 'yolo_classifier'（后者只是类型用）。
  • bash_classifier 仅 ant 可用，用 feature('BASH_CLASSIFIER') 门控，让该字符串能从外部构建里被 tree-shake（写在 excluded-strings.txt）。

2.3 哪些是"后台"被放弃的？

不在白名单里的 source——摘要、标题生成、prompt 建议、各种分类器——在 529 时立即抛 CannotRetryError（见 [6]）。它们的失败用户根本看不到，没必要为它们放大网关压力。

三、判定函数 shouldRetry529

function shouldRetry529(querySource: QuerySource | undefined): boolean {
  // undefined → 重试（对未打标的调用路径采取保守策略）
  return (
    querySource === undefined || FOREGROUND_529_RETRY_SOURCES.has(querySource)
  )
}
输入返回含义
undefined（未打标）true保守：拿不准就重试，宁可多放大也别误伤前台
白名单内true前台请求，重试
其他（后台）false立即放弃
为什么 undefined → 重试：调用方没传 querySource 时无法判断前后台，保守策略是当成前台对待。代价是偶尔多重试一次，但不会误杀用户在等的请求。

这个函数在 [6] 的 529 处理里被调用：is529Error(error) && !shouldRetry529(options.querySource) → 后台 drop。

四、速记口诀

  • 四常量：MAX_RETRIES=10 · FLOOR=3000 · MAX_529=3 · BASE_DELAY=500（唯一 export）。
  • 白名单本质：用户在阻塞等待 = 前台 = 529 时重试；后台（摘要/标题/分类器）= 立即放弃。
  • 为什么：529 容量级联期，每次重试 3-10× 网关放大，后台失败用户看不到。
  • shouldRetry529：白名单内 或 undefined → 重试；其余 → drop。
