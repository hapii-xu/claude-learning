Provider 分流：OpenAI / Gemini / Grok 兼容层

这是 queryModel 里一个提前 return 的分叉（claude.ts:1621-1662）。Claude Code 支持 7 个 provider，其中 OpenAI/Gemini/Grok 走的是兼容层（把第三方 API 格式转成 Anthropic 内部格式）。当检测到这三者之一时，queryModel 把活儿交给对应的 queryModel* 函数，然后直接 return——本文件后面那一大堆 Anthropic 专属逻辑（betas、thinking、cache breakpoints、流式解析）全部跳过。

一、分流点的位置：为什么是这里

// OpenAI 兼容的 provider：在共享的预处理（消息归一化、工具过滤、
// 媒体剔除）之后、Anthropic 专属逻辑（betas、thinking、caching）之前，
// 交给 OpenAI 适配层处理。
if (getAPIProvider() === 'openai') { ... return }

注释点明了分流点的精心选址：

┌─────────────────────────────────────────┐
│ 共享预处理（所有 provider 都需要）         │
│  · 工具过滤 [4][5]                         │
│  · 消息归一化 + 清洗 [6]                    │
│  · 媒体剔除 [6]                            │
├─────────────────────────────────────────┤  ← ★ 分流点在这里
│ Anthropic 专属（只有第一方/Bedrock 需要）  │
│  · betas / thinking / cache breakpoints   │
│  · 流式 SSE 解析 [10]-[16]                  │
└─────────────────────────────────────────┘

把分流点放在两段之间，让三个兼容层能复用上面所有通用的消息/工具预处理，又不必趟下面那堆 Anthropic 专属的水。这是很经济的设计——不在每个兼容层里重复实现归一化、配对修复、媒体剔除。

二、三个分流分支

2.1 OpenAI（1624-1637）

if (getAPIProvider() === 'openai') {
  const { queryModelOpenAI } = await import('./openai/index.js')
  yield* queryModelOpenAI(
    messagesForAPI,
    systemPrompt,
    tools,            // ← 注意：完整 tools，不是 filteredTools
    signal,
    options,
  )
  return
}

2.2 Gemini（1639-1650）

if (getAPIProvider() === 'gemini') {
  const { queryModelGemini } = await import('./gemini/index.js')
  yield* queryModelGemini(
    messagesForAPI,
    systemPrompt,
    filteredTools,    // ← filteredTools
    signal,
    options,
    thinkingConfig,   // ← Gemini 多传了 thinkingConfig
  )
  return
}

2.3 Grok（1652-1662）

if (getAPIProvider() === 'grok') {
  const { queryModelGrok } = await import('./grok/index.js')
  yield* queryModelGrok(
    messagesForAPI,
    systemPrompt,
    filteredTools,    // ← filteredTools
    signal,
    options,
  )
  return
}

三、⭐ 关键差异：tools vs filteredTools

三个分支传的工具列表不一样，这是本节最值得记住的细节：

Provider传的工具原因
OpenAI完整 tools要在客户端模拟 Anthropic 的动态工具加载
GeminifilteredTools直接用过滤后的工具集
GrokfilteredTools同上

3.1 OpenAI 为何要完整工具池

// OpenAI 在客户端模拟 Anthropic 的动态工具加载。它需要完整的工具池，
// 以便 SearchExtraToolsTool 可以搜索那些被有意从上面的初始 API 工具列表
// 里过滤掉的延迟 MCP 工具。

回顾 [4][5]：Anthropic 路径下，延迟工具被服务端机制管理，客户端只发核心工具 + SearchExtraTools。但 OpenAI 协议没有这套服务端能力，所以兼容层得自己在客户端实现整个"工具搜索/延迟加载"流程。

要模拟它，SearchExtraToolsTool 必须能"看到"所有延迟工具（才能搜出来），因此 OpenAI 分支必须拿到完整 tools——而不是已经被剔掉延迟工具的 filteredTools。

对比 [5] 里 toolToAPISchema 的同款思路：真正发送可以是过滤后的，但"能搜到什么"的上下文必须是完整的。OpenAI 把这条原则贯彻到整个查询。

3.2 Gemini/Grok 为何用 filteredTools

这两个兼容层的工具处理策略不同，直接消费过滤后的工具集即可（具体实现见各自兼容层 docs）。Gemini 还额外需要 thinkingConfig（因为它有自己的思考预算映射）。

四、yield* 委托：流如何穿透

yield* queryModelOpenAI(...)
return

yield*（委托生成器）的作用：把 queryModelOpenAI 这个生成器吐出的每一个值，原样转发给 queryModel 的消费者。也就是说兼容层产出的 StreamEvent/AssistantMessage/错误消息（同样的三类，见 [0]）会透明地流到上层——上层完全不知道背后是 OpenAI 还是 Anthropic。

这正是 CLAUDE.md 里说的"流适配器模式：将第三方 API 格式转为 Anthropic 内部格式，下游代码完全不改"。yield* + return 就是这个"完全不改"的实现机制。

五、动态 import 的好处

const { queryModelOpenAI } = await import('./openai/index.js')

和 [5] 的 cachedMicrocompact 同理：三个兼容层用动态导入，只在真正用到对应 provider 时才加载其模块。第一方用户（绝大多数）永远不会把 OpenAI/Gemini/Grok 的代码加载进内存——契合 CLAUDE.md 强调的代码分割 / 按需加载（减小 RSS）。

六、谁不在这里：firstParty 与 Bedrock

getAPIProvider() 的 7 个值里，这里只拦截 3 个。剩下的：

Provider是否在此分流走向
firstParty（Anthropic 直连）否继续往下（[8]+）
bedrock（AWS）否继续往下，但内部有多处 getAPIProvider()==='bedrock' 特判
vertex、foundry否继续往下（走 Anthropic SDK 兼容路径）
openai / gemini / grok是兼容层 + return

所以这之后的所有逻辑（[8]~[16]）都默认运行在"Anthropic SDK 协议"下（含 Bedrock/Vertex 这类仍用 Anthropic 消息格式的网关）。

速记口诀
