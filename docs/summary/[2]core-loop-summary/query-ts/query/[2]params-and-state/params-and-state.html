输入与状态契约：QueryParams 与 State

query() 的输入是不可变的 QueryParams（13 字段），其中 toolUseContext.langfuseTrace 决定 ownsTrace、querySource 决定持久化与 trace 归属；queryLoop 跨迭代携带的可变状态是 State（10 字段，重点 transition——让测试能断言恢复路径）。

一、QueryParams：不可变的输入契约（13 字段）

export type QueryParams = {           // query.ts:284-317
  messages: Message[]
  systemPrompt: SystemPrompt
  userContext: { [k: string]: string }
  systemContext: { [k: string]: string }
  canUseTool: CanUseToolFn
  toolUseContext: ToolUseContext
  fallbackModel?: string
  querySource: QuerySource
  maxOutputTokensOverride?: number
  maxTurns?: number
  skipCacheWrite?: boolean
  taskBudget?: { total: number }
  deps?: QueryDeps
}
字段作用与本系列的关联
messages对话历史，每轮把新回复 + 工具结果追加进副本query() 透传
systemPrompt基础系统提示（不含 CLAUDE.md / git 动态上下文）—
userContext用户侧动态上下文（CWD/shell/日期），调 API 前拼到 messages 前—
systemContext系统侧动态上下文（CLAUDE.md / git status），拼到 systemPrompt 后—
canUseTool工具权限回调，每次工具调用前判定是否放行—
toolUseContext工具执行上下文：agentId / mainLoopModel / 权限模式 / AppState / langfuseTrace⭐ 决定 [3] 的 ownsTrace
fallbackModel?兜底模型名，主模型限流/不可用时切它重试queryLoop 降级用
querySource来源标签（repl_main_thread / agent:*）⭐ 持久化 + trace 归属 + 埋点
maxOutputTokensOverride?输出 token 上限覆盖值（限流恢复 / 超窗重试用）进入 State 副本
maxTurns?循环最大迭代轮数，未设为 ∞（防 agent 无限工具调用）queryLoop 终止判定
skipCacheWrite?跳过 prompt cache 写入（仅读），避免大查询污染命中率—
taskBudget?API task_budget（beta），total 是整轮预算queryLoop 每轮算 remaining
deps?依赖注入（microcompact / autocompact 等），测试替 mock未提供时用 productionDeps()
关键点：query() 几乎原样透传 QueryParams——它唯一对入参做的修改，是在有 trace 时浅拷贝出 paramsWithTrace，把 trace 挂进 toolUseContext（见 [3]）。除此之外，params 在整个回合里不可变。

二、State：循环内部的可变状态（10 字段）

State 不是 query() 的入参，而是 queryLoop 跨迭代携带的可变状态。query() 不直接碰它，但理解它有助于看清「为什么 query 的善后逻辑能放在外层」。

type State = {                              // query.ts:322-335
  messages: Message[]
  toolUseContext: ToolUseContext
  autoCompactTracking: AutoCompactTrackingState | undefined
  maxOutputTokensRecoveryCount: number
  hasAttemptedReactiveCompact: boolean
  maxOutputTokensOverride: number | undefined
  pendingToolUseSummary: Promise<ToolUseSummaryMessage | null> | undefined
  stopHookActive: boolean | undefined
  turnCount: number
  // 上一次迭代为什么继续。首次迭代为 undefined。
  transition: Continue | undefined
}
字段含义
messages当前迭代要发给 API 的消息（每轮重建副本）
toolUseContext工具上下文（唯一会被循环内重赋值的 params 衍生物）
autoCompactTracking自动压缩的跨边界追踪状态
maxOutputTokensRecoveryCount续写恢复已用次数（对照 [1] 的 LIMIT=3）
hasAttemptedReactiveCompact是否已尝试过反应式压缩（防死亡螺旋的守卫）
maxOutputTokensOverride当前输出上限覆盖（escalate 时升级）
pendingToolUseSummary上一轮 haiku 工具摘要的 Promise（本轮流式期间解析）
stopHookActivestop hook 是否激活
turnCount已迭代轮数（对照 maxTurns）
transition⭐ 上一次迭代「为什么继续」（首轮 undefined）
transition 为什么重要：它记录上一次迭代的续轮原因（next_turn / collapse_drain_retry / reactive_compact_retry / …）。存在的唯一目的：让测试可以断言恢复路径已触发，而不必去检查消息内容。这是「把控制流显式化、便于测试」的典型设计。

三、params 不可变 vs state 副本更新

这是整个 query.ts 的核心设计模式：

QueryParams (不可变 · 外部契约) │ 浅拷贝挂 trace ▼ paramsWithTrace │ 首轮初始化 ▼ State #0 ──"state = { ...next }"──► State #1 ──► State #2 ... （每次迭代整体替换，而非 10 个独立赋值）
  • QueryParams 不可变：作为外部契约，整轮只读，保证 query() 的善后逻辑（trace 归属、命令收尾）能基于稳定的输入推理。
  • State 整体副本更新：queryLoop 每次迭代结束用 state = { ...变更 } 整体替换，而非 10 个独立赋值——新增续轮路径时只改一处构造，不会漏字段、不污染多个续轮点。
这种「不可变入参 + 整体副本状态」的分工，让 query() 能把生命周期善后（trace、内存、命令）安全地放在循环之外的 try/finally 里——因为它依赖的全是不可变的 params 与最终的 terminal，不受循环内部状态变迁影响。State 的逐字段细节见 queryLoop/[2]params-state-deps。
