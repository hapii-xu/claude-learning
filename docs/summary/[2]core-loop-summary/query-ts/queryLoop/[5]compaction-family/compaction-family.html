压缩家族 snip → micro → collapse

autocompact 之前的三种瘦身手段及固定执行顺序 —— snipCompact（HISTORY_SNIP）、microcompact（清理原始字符串、延迟边界消息）、contextCollapse（读取时投影，放 autocompact 前以保细粒度）。三者互补不互斥。

query.ts:802–895 互补不互斥 保留细粒度 压缩家族 ②
  • 一、snipCompact
  • 二、microcompact
  • 三、contextCollapse
  • 四、为什么这个顺序
执行顺序：snipCompact → microcompact → contextCollapse →（autocompact，见 [6]）。共同目标：在调用昂贵的 autocompact 之前先用便宜手段把 token 降下来，让它尽量成为 no-op，从而保留细粒度上下文而非单一摘要。

一、snipCompact（HISTORY_SNIP）

let snipTokensFreed = 0
if (feature('HISTORY_SNIP')) {
  const snipResult = snipModule!.snipCompactIfNeeded(messagesForQuery)
  messagesForQuery = snipResult.messages
  snipTokensFreed = snipResult.tokensFreed
  if (snipResult.boundaryMessage) yield snipResult.boundaryMessage
}
  • 在 microcompact 之前应用（两者可能都跑，不互斥）。
  • snipTokensFreed 是关键产物：传给下游 autocompact / 阻塞检查，让阈值反映 snip 移除的量。
  • 为何必须显式传：tokenCountWithEstimation 看不到 snip 效果——它从受保护尾部的 assistant 读 usage，尾部在 snip 中不变。

二、microcompact

const microcompactResult = await deps.microcompact(messagesForQuery, toolUseContext, querySource)
messagesForQuery = microcompactResult.messages
if (microcompactResult.clearedToolUseIds?.length) {     // ⭐ 内存治理
  for (const id of microcompactResult.clearedToolUseIds) replacements.delete(id)
}
const pendingCacheEdits = feature('CACHED_MICROCOMPACT')
  ? microcompactResult.compactionInfo?.pendingCacheEdits : undefined
要点说明
clearedToolUseIds 释放工具结果内容换成清理消息后，删 replacements 里的原始字符串
pendingCacheEdits边界消息延迟到 API 响应后才发，用真实 cache_deleted_input_tokens（见 [9]）
feature 门控让被排除的字符串从外部构建中消失
microcompact 与工具结果预算（[4]）干净组合：缓存 MC 只看 tool_use_id，从不看内容，所以内容替换对它不可见。

三、contextCollapse.applyCollapsesIfNeeded

if (feature('CONTEXT_COLLAPSE') && contextCollapse) {
  const collapseResult = await contextCollapse.applyCollapsesIfNeeded(messagesForQuery, toolUseContext, querySource)
  messagesForQuery = collapseResult.messages
}
  • 放在 autocompact 之前：若折叠已让我们低于阈值，autocompact 成 no-op，保留细粒度上下文。
  • 不 yield 任何东西：折叠视图是对完整历史的读取时投影，摘要消息住在 collapse 存储里、不在 REPL 数组中。
  • 这正是折叠能跨轮次持久的原因：projectView() 每次入口重放提交日志；轮次内视图通过续轮点（[15]）的 state.messages 流转。

四、为什么是这个顺序

便宜、保细粒度 ───────────────────────→ 昂贵、变摘要 snip → microcompact → collapse → autocompact

越靠前越便宜、越能保留原始信息；越靠后越激进。把激进手段放最后，让前面的便宜手段有机会先把 token 降到阈值之下，autocompact 自然 no-op。

速记口诀

  • 顺序：snip → micro → collapse →（autocompact），互补不互斥。
  • snipTokensFreed 必须显式向下传，因陈旧 usage 看不到 snip 效果。
  • microcompact 清原始字符串（内存）；pendingCacheEdits 延迟边界消息。
  • collapse 是读取时投影、不 yield、跨轮次持久，放 autocompact 前保细粒度。
