阻塞上限 与预测式压缩

发请求前最后两道 token 关卡 —— 硬阻塞上限检查（auto-compact 关闭时撞限返回 blocking_limit，以及各恢复路径为何要跳过以防死锁/饿死），与预测式 autocompact（用 estimateMaxTurnGrowth + 不含缓冲的窗口估算本轮增长是否超窗）。

一、门控上提：collapseOwnsIt / mediaRecoveryEnabled

const mediaRecoveryEnabled = reactiveCompact?.isReactiveCompactEnabled() ?? false
mediaRecoveryEnabled 每轮一次上提：扣留（流循环内 [9]）与恢复（之后 [12]）必须用同一个值。CACHED_MAY_BE_STALE 可能在 5–30s 流期间翻转，若扣留了却不恢复就会吃掉消息。

二、硬阻塞上限检查 → blocking_limit

if (!compactionResult && querySource !== 'compact' && querySource !== 'session_memory' &&
    !(reactiveCompact?.isReactiveCompactEnabled() && isAutoCompactEnabled()) && !collapseOwnsIt) {
  const estimatedTokens = tokenCountWithEstimation(messagesForQuery) - snipTokensFreed
  const { isAtBlockingLimit } = calculateTokenWarningState(estimatedTokens, ...)
  if (isAtBlockingLimit) {
    yield createAssistantAPIErrorMessage({ content: PROMPT_TOO_LONG_ERROR_MESSAGE, error: 'invalid_request' })
    return { reason: 'blocking_limit' }
  }
}

仅在 auto-compact 关闭时生效——保留空间让用户仍能手动 /compact。几处跳过各防一种灾难：

跳过条件原因
compactionResult 刚发生压缩已验证在阈值下，但 tokenCountWithEstimation 会读到陈旧的压缩前 input_tokens
compact / session_memory继承完整对话的 fork agent，在此阻塞会死锁（compact agent 需运行才能降 token）
reactive 启用 + 允许自动压缩preempt 合成错误在 API 调用前返回，reactive 永远看不到可响应的 PTL
collapseOwnsItcollapse 的 recoverFromOverflow 在真实 413 排空已分阶段 collapse，此处合成 preempt 会饿死它
estimatedTokens 减去 snipTokensFreed——否则在 snip 已降到阈值下、但陈旧用量仍高于阻塞限制的窗口中会错误阻塞。

三、预测式 autocompact

if (!compactionResult && isAutoCompactEnabled()) {
  const currentTokens = tokenCountWithEstimation(messagesForQuery) - snipTokensFreed
  const estimatedGrowth = estimateMaxTurnGrowth(model)
  const predictiveThreshold = getEffectiveContextWindowSize(model) - estimatedGrowth
  if (currentTokens > predictiveThreshold) {
    const predictiveResult = await deps.autocompact(...)
    if (predictiveResult.compactionResult) {
      messagesForQuery = buildPostCompactMessages(predictiveResult.compactionResult)
      snipTokensFreed = 0
    }
  }
}

主动 autocompact（[6]）基于"当前已超阈值"触发；预测式则估算本轮增长是否会超窗：

  • estimateMaxTurnGrowth(model)：本轮最多可能新增多少 token。
  • getEffectiveContextWindowSize(model)：不带 autocompact 缓冲的窗口——避免与 getAutoCompactThreshold（已减缓冲）重复预留。
  • 命中则压缩，并把 snipTokensFreed 归零（已用新的压缩后消息）。

速记口诀

  • 门控上提：mediaRecoveryEnabled 每轮一次，保证扣留与恢复同值。
  • blocking_limit：仅 auto-compact 关闭；compact/session_memory/reactive/collapse 跳过以防死锁/饿死；减 snipTokensFreed 防误阻塞。
  • 预测式压缩：用增长估算 + 不含缓冲的窗口，提前压缩防本轮爆窗。
