# Pending 单例 · argv 预解析与消费

> `src/main.tsx:718–763` 定义了三套「暂存-消费」模块单例：`_pendingConnect`（DIRECT_CONNECT）、`_pendingAssistantChat`（KAIROS）、`_pendingSSH`（SSH_REMOTE）。它们是 main.tsx 最独特的设计模式——`main()` 写入，主 action 消费。

---

## 一、为什么需要模块单例？

Commander 的 `parseAsync` 会自动分发 argv 到对应 `.action(handler)`，**只给两个参数**：`(prompt, options)`。

```ts
program.argument('[prompt]', '...').action(async (prompt, options) => {
  // 只能访问 prompt 和 options
  // main() 无法传入额外数据
});
```

> **问题**：`main()` 需要预解析某些 argv（cc:// URL / `assistant` / `ssh`），但没法传给 action。
> **解决**：模块级变量是唯一跨函数通信的方式。

## 五、为什么不用函数参数？

### 5.1 Commander 的限制

Commander.js 设计时就没有"额外参数"机制：

```ts
// ❌ 不支持
program.action((prompt, options, extraData) => { ... });

// ✅ 只支持
program.action((prompt, options) => { ... });
```

### 5.2 模块级变量的唯一性

| 方案 | 可行性 | 问题 |
|---|---|---|
| 函数参数 | ❌ | Commander 不支持 |
| 全局变量 | ✅ | 污染全局命名空间 |
| 环境变量 | ✅ | 需要序列化复杂对象 |
| 模块级 const | ✅ | **当前方案**——作用域限于此文件 |

## 六、常见问题 FAQ

> **Q：为什么是 `const` + 可变内部对象，而不是 `let`？**

A：JavaScript 对象引用不变，内容可变。`const` 保证引用不会被意外重新赋值，而内部字段可写入。这与 TypeScript 的类型声明一致。

> **Q：为什么不存到 settings.json？**

A：这些是**单次会话级别的临时数据**，不需要持久化。写入 settings 会污染配置，且读写有 I/O 开销。

> **Q：如果 feature 关闭，这些单例是什么？**

A：`undefined`。定义时的 ternary 直接返回 `undefined`，后续 `main()` 写入前会检查 feature，避免写入不存在的对象。
