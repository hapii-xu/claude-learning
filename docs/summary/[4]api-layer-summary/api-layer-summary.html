一、双层架构全貌

API 通信层被拆成两层：编排层管理多轮对话状态与工具执行，执行层负责单次 API 调用的参数组装与流事件处理。

编排层 src/query.ts

  • query() — 对外入口，初始化上下文
  • queryLoop() — 多轮循环驱动核心
  • 管理对话历史 messages[]
  • 调用 deps.callModel() 委托执行层
  • 收集并解析 AssistantMessage
  • 执行 tool_use 块 → 追加 tool_result
  • auto-compact 检查（413 / token 阈值）
  • Token Budget 管理

执行层 src/services/api/claude.ts

  • queryModel() — 单次请求入口（:1049）
  • queryModelWithStreaming() — 流式调用（:775）
  • 组装完整请求参数（system/messages/tools/betas）
  • 路由到目标 Provider
  • 处理 5 种流式事件
  • yield StreamEvent 回编排层
  • 计算 usage / cost 写入 bootstrap state
💡
依赖注入设计：queryLoop() 通过 deps.callModel 间接调用执行层，使测试时可注入 mock 实现。真实路径是 deps.callModel → queryModelWithStreaming → queryModel。

二、一次 API 调用的完整生命周期

从 queryLoop() 的一次迭代开始，到工具结果写入下一轮消息。

1
queryLoop() 迭代开始
src/query.ts :430 — 调用 fetchSystemPromptParts() 获取三源汇流的 system prompt
2
deps.callModel(messages, systemPrompt, tools, …)
委托给 queryModelWithStreaming()，传入当前轮次消息、系统提示、工具列表
3
queryModel() 组装请求参数
buildSystemPromptBlocks() · addCacheBreakpoints() · getMergedBetas() · allTools[] · max_tokens 解析
4
anthropic.beta.messages.create() 流式调用
Anthropic SDK 发起 HTTP stream，开始接收 BetaRawMessageStreamEvent
5
stream event loop（5 种事件循环处理）
message_start → content_block_start → content_block_delta → content_block_stop → message_delta
6
yield AssistantMessage（含所有 content blocks）
文本/思考/tool_use 块全部归一化，回传 queryLoop()
7
queryLoop() 判断分支
有 tool_use → runTools() 或 StreamingToolExecutor 执行工具 → 追加 tool_result → 下一轮
无 tool_use → 返回 { reason: 'completed' }

三、必须掌握的六大核心概念

1
请求参数组装（buildApiRequestParams）

核心位置：src/services/api/claude.ts paramsFromContext() 闭包（约 line 1634）

// 最终发送给 API 的请求参数结构
{
  model: normalizeModelStringForAPI(options.model),
  system: buildSystemPromptBlocks(systemPrompt, isSkipCacheWrite),  // 含 cache_control
  messages: addCacheBreakpoints(normalizedMessages, skipCacheWrite),  // 末尾插断点
  tools: allTools,        // 内置工具在前，MCP 工具追加末尾
  tool_choice: options.toolChoice,
  betas: filteredBetas,   // 过滤空字符串后的功能标志
  max_tokens: maxOutputTokens,         // 首轮 8K，截断后升 64K
  thinking: { type: 'enabled', budget_tokens: N },  // interleaved thinking 时
  temperature: 1,       // 仅在 thinking 关闭时
  metadata: getAPIMetadata(),
  speed: 'fast',         // 快速模式
  context_management: {},  // 上下文管理 beta
  output_config: { effort, task_budget, format },
  ...extraBodyParams,      // env 注入的额外参数
}
📌
system 字段是 string[]，每个元素是独立 system block，支持 cache_control。静态前缀块有 5min 或 1h TTL，动态块（如 mcp_instructions）每轮重算。
2
Beta 标志动态管理（getMergedBetas + 粘性锁存）

核心文件：src/utils/betas.ts（getMergedBetas）、src/constants/betas.ts

Beta 分三层叠加：

Layer 1: getModelBetas()       ← 模型级固定 beta（按模型版本确定，memoized）
Layer 2: isAgenticQuery betas  ← 代理查询时追加（claude-code / interleaved-thinking 等）
Layer 3: 动态粘性锁存 (latch)  ← 运行时条件触发，写入 bootstrap state 不可逆
Beta 名称 Header 值 启用条件
Claude Codeclaude-code-20250219所有代理查询
Interleaved Thinkinginterleaved-thinking-2025-05-14思考模式
Context 1Mcontext-1m-2025-08-071M 上下文开关

🔒
粘性锁存（latch）：fast-mode、AFK-mode、cache-editing 等 beta 一旦激活写入 bootstrap state，当前会话不可逆，避免请求间反复切换导致缓存失效。
3
流式事件处理（5 种事件类型）

核心位置：src/services/api/claude.ts line 2088–2387

事件 触发时机 主要动作
message_start 流开始 初始化 partialMessage，记录 TTFT，更新初始 usage（含 cache_read_input_tokens）
content_block_start 新内容块 按 type 初始化（tool_use/text/thinking），input/text 设为空串
content_block_delta 增量内容 追加 text_delta→ text map，input_json_delta→ tool input，thinking_delta→ thinking
content_block_stop 块结束 join text map（O(n) 非 O(n²)），归一化 content block，yield AssistantMessage
message_delta 流结束 更新最终 usage，计算 cost，记录 stop_reason

文本累积优化（line 2243, 2314）—— 避免 O(n²) 字符串拼接：

// delta 阶段：O(1) 追加到 Map
Map<blockIndex, string[]>.push(delta)
// stop 阶段：O(n) 一次 join
map.get(index).join('')
⚡
在长文本输出时，这个优化效果显著——逐 delta 字符串拼接的 O(n²) 在几千个 delta 时会导致明显卡顿。
4
Tool Use 块的收集与执行流程

收集阶段（执行层，流内拼装）：

// 1. content_block_start
{ type: 'tool_use', id, name }  → contentBlock = { type, id, name, input: '' }

// 2. content_block_delta（逐片追加 JSON 字符串）
{ type: 'input_json_delta', partial_json }  → contentBlock.input += partial_json

// 3. content_block_stop（解析完整 JSON）
JSON.parse(contentBlock.input)  → 完整 tool input
yield AssistantMessage（含 tool_use 内容块）

执行阶段（编排层，queryLoop 处理）：

非流式工具 — runTools()

串行执行所有 tool_use 块，适用于大多数内置工具。执行结果汇总后批量追加。

src/services/tools/toolOrchestration.ts

流式工具 — StreamingToolExecutor

边收流边并行执行，减少等待时间。适用于支持实时输出的工具（如 BashTool）。

src/services/tools/StreamingToolExecutor.ts
🔐
执行前经过 canUseTool() 权限检查 → 弹框确认或 bypassPermissions 自动通过。
5
单一 addCacheBreakpoints 策略

核心位置：src/services/api/claude.ts line 3241

// 普通请求：cache_control 插入最后一条消息
messages[last]

// skipCacheWrite 请求（fork 子代理）：插入倒数第二条
// 避免污染父会话的 KV cache 状态
messages[last-2]

系统提示词缓存策略（buildSystemPromptBlocks line 3392）：

块类型cache_controlTTL说明
静态前缀块 有 5min 或 1h should1hCacheTTL() latch 控制避免频繁切换
动态块（mcp_instructions 等） 无 — 每轮重算，不缓存
MCP 工具列表为空 + global cache scope: global 跨会话 system prompt 跨会话共享 KV cache

7 种 Provider 及选择逻辑

核心位置：src/utils/model/providers.ts → getAPIProvider()

选择优先级（高→低）：

1. settings.modelType ('openai'/'gemini'/'grok')    ← UI 持久化设置
2. CLAUDE_CODE_USE_BEDROCK / VERTEX / FOUNDRY       ← 企业部署环境变量
3. CLAUDE_CODE_USE_OPENAI / GEMINI / GROK           ← 兼容层开关
4. 默认：firstParty（Anthropic API 直连）

详见 七种 Provider 节。

四、七种 Provider 详解

所有非 firstParty 的 Provider 均采用流适配器模式：将第三方 API 格式转为 Anthropic 内部格式，下游代码零改动。

1
firstParty 默认

Anthropic SDK 直连，全功能：thinking、advisor、cache、betas、context management 完整支持。

src/services/api/claude.ts

2
bedrock AWS

betas 通过 extraBodyParams.output_config 传递（非 header）。功能集受限，部分 beta 不可用。

需配置 AWS 凭证环境变量

3
vertex GCP

Google Cloud。支持 Web Search beta，不支持 Structured Outputs。需配置 GCP 服务账户。

CLAUDE_CODE_USE_VERTEX=1

4
foundry 企业

企业自托管。支持 ISP（Integrated System Prompt）+ context management。

CLAUDE_CODE_USE_FOUNDRY=1

5
openai 兼容层

适配 OpenAI Chat Completions API（Ollama / DeepSeek / vLLM）。含消息/工具格式转换 + DeepSeek thinking 支持。

src/services/api/openai/index.ts

6
gemini 兼容层

Google Gemini API。独立模型映射优先级：GEMINI_MODEL > GEMINI_DEFAULT_* > 原样返回。

src/services/api/gemini/index.ts

7
grok 兼容层

xAI Grok API。消息格式转换，自定义模型映射。

src/services/api/grok/index.ts
🔌
流适配器模式：OpenAI / Gemini / Grok 均在内部将各自格式转为 BetaRawMessageStreamEvent，queryModelWithStreaming() 及以上层看不到任何差异——这就是为什么添加新 Provider 只需实现适配器，核心循环一行代码都不用改。

五、应该了解的辅助机制

A. Usage / Cost 双轨追踪

updateUsage()（line 3102）— 流内更新，防止 0 值覆盖缓存命中数：

message_start  → 初始 usage（含 cache_read）
content_block_stop → 中间状态
message_delta  → 最终 usage（stop 后精确值）

成本计算（line 2377）：

const costUSD = calculateUSDCost(model, usage)
addToTotalSessionCost(costUSD, usage, model)
// → 写入 bootstrap STATE 的 sessionCost

accumulateUsage()（line 3171）跨轮次叠加，用于 session 级别总成本统计。

B. Max Tokens 两段式升级

首轮：max_tokens = CAPPED_DEFAULT_MAX_TOKENS
      = 8,000  ← 节省 KV cache slot
  ↓
stop_reason === 'max_tokens'（输出被截断）
  ↓
重试：max_tokens = ESCALATED_MAX_TOKENS
         = 64,000
最多重试 3 次
💡
90% 的请求不会触发截断，首轮 8K 极大降低了平均 KV cache 占用和成本。

C. 流超时与重试

  • 流空闲超时：90s（CLAUDE_STREAM_IDLE_TIMEOUT_MS 可覆盖）
  • API 返回 413（请求过大）→ reactiveCompact.ts 事后压缩，重试
  • Fallback 模型：配置后可在主模型失败时切换（一次）
  • 流式错误（网络中断）→ 重建连接，最多 N 次

D. 缓存断裂检测

src/services/api/promptCacheBreakDetection.ts

监测预期应命中 cache 但实际发生 cache miss 的情况，上报遥测。帮助识别：

  • 系统提示词意外变化
  • 消息格式不稳定导致断点偏移
  • MCP 工具描述频繁更新

E. QueryEngine — 高层编排器

src/QueryEngine.ts 是包裹 query() 的高层编排器，被 REPL screen 直接使用：

  • 管理会话级对话状态
  • 触发 自动压缩（compaction）
  • 文件历史快照（file history snapshot）
  • Attribution 归因标记
  • Turn-level 记账（每轮 token/cost 统计）
  • 向 REPL 推送 StreamEvent 更新 UI

七、与其他模块的关联

已学模块 与 API 通信层的关联点
上下文工程 fetchSystemPromptParts 三源汇流的产物作为 system 参数传入 queryModel
工具系统 allTools 参数来自 useMergedTools（内置 + MCP），CORE_TOOLS 白名单决定 deferred 工具
状态管理 bootstrap STATE 存储 sessionCost、model overrides；AppState 存储消息历史
MCP 协议 MCP 工具追加末尾保护 prompt cache；DANGEROUS_uncachedSystemPromptSection 每轮重算 MCP 工具描述
安全与权限 工具执行前调用 canUseTool()；bypassPermissions 直接通过不弹框
核心循环 queryLoop 是核心循环的数据源，QueryEngine 封装多轮状态供 REPL screen 消费
入口 / Bootstrap Bootstrap state 的 sessionCost、fastMode latch、model overrides 均由 API 层写入

八、验证清单

完成本篇学习后，你应该能够回答以下问题：

说出请求组装中 system / messages / tools / betas 各字段的来源（三源汇流 → buildSystemPromptBlocks / addCacheBreakpoints / useMergedTools / getMergedBetas）
解释为什么 max_tokens 首轮是 8K 而不是直接用 64K（节省 KV cache slot，90% 请求不被截断）
描述 5 种流式事件的处理顺序和各自的关键动作（message_start → content_block_start → delta → stop → message_delta）
解释文本累积为什么用 Map<index, string[]> 而不是直接字符串拼接（避免 O(n²) 的拼接开销）
说出 7 种 Provider 的选择优先级和各自的主要差异（settings.modelType → 环境变量企业级 → 兼容层开关 → firstParty）
解释 Beta 粘性锁存（latch）的作用——为什么 fast-mode 等 beta 一旦激活就不可逆（防止会话内反复切换破坏 KV cache）
描述 skipCacheWrite 模式下 cache breakpoint 插入位置的变化（从最后一条改为倒数第二条，避免污染父会话 cache）
