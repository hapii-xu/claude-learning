一、概念与整体架构

Skill 是存放在 Markdown 文件中的可复用工作流提示词模板。它告诉模型应该如何思考这类任务、可以调用哪些工具、按什么顺序执行步骤。与工具（Tool）不同，Skill 不给模型新的底层能力，而是提供预定义的工作流程。

两种执行模式

内联模式（默认）

  • 提示词注入当前对话
  • 模型能看到整个对话历史
  • 适合交互式、需要上下文的任务
  • 示例：/commit（需要看到改了什么）

分叉模式（context: fork）

  • 创建独立子代理，独立 token 预算
  • 子代理有独立对话状态
  • 适合自包含任务（不需要历史上下文）
  • 示例：/generate-tests（独立工作）

目录结构

.claude/
└── skills/
    ├── my-skill/
    │   ├── SKILL.md          ← 必须（技能定义）
    │   ├── helper.sh         ← 可选（辅助脚本）
    │   └── schema.json       ← 可选（数据结构定义）
    └── another-skill/
        └── SKILL.md

~/.claude/
└── skills/
    └── global-skill/
        └── SKILL.md          ← 用户级全局 Skill

调用方式

# 用户在 REPL 中直接输入
/my-skill arg1 arg2

# 模型自主调用（通过 SkillTool）
# 模型会根据 when-to-use 字段决定何时调用

二、SKILL.md 完整 Frontmatter 字段字典

每个 Skill 的 SKILL.md 文件由 YAML frontmatter（元数据）+ 正文（提示词内容）两部分组成。

---
# ← YAML frontmatter（元数据）
description: 这个 Skill 做什么
allowed-tools: [BashTool, Read]
# ...
---

# ← 正文（提示词内容）
请按以下步骤执行...
字段类型默认值说明
descriptionstring从正文第一行提取核心字段，显示在 / 命令列表，也是模型看到的工具描述
namestring目录名覆盖显示名称（不影响调用名，调用名始终是目录名）
argument-hintstring—UI 输入提示，如 <file-path> [options]
argumentsstring \| string[][]命名参数列表，定义后可用 $ARG1、$ARG2 引用
when-to-usestring—告诉模型何时主动调用此 Skill（关键！）
allowed-toolsstring[][]执行时自动允许的工具（不弹权限确认框）
modelstring继承当前模型覆盖执行模型，如 claude-opus-4-8
effortstring \| number继承努力程度：low / medium / high 或数字
context"fork"—（内联）设为 fork 使用独立子代理执行
agentstring—指定执行此 Skill 的 Agent 类型名
user-invocablebooleantruefalse = 从 / 命令列表隐藏（仅供模型调用）
disable-model-invocationbooleanfalsetrue = 只注入提示词，不触发模型调用
versionstring—语义版本号，如 1.2.0
pathsstring—gitignore 格式的路径模式，只有触碰匹配文件时才激活
hooksobject—内联 Hook 配置，格式与 settings.json 相同
shellobject—Shell 执行配置（控制 ! 命令的执行方式）

三、参数注入语法

1
$ARGUMENTS — 全量参数字符串
 
---
description: 提交代码
---

请用以下 commit message 提交当前暂存的代码：

**$ARGUMENTS**

如果 $ARGUMENTS 为空，请先询问用户想要什么样的 commit message。

调用：/commit fix: 修复登录验证问题

$ARGUMENTS 展开为：fix: 修复登录验证问题

2
arguments + $ARG1 / $ARG2 — 命名位置参数
 
---
description: 对比两个文件的差异
arguments: [file1, file2]
argument-hint: "<file1> <file2>"
allowed-tools: [Read, BashTool]
---

请对比以下两个文件的差异：

- 文件 1：$ARG1
- 文件 2：$ARG2

首先读取两个文件，然后分析差异并给出建议。

调用：/diff src/old.ts src/new.ts

  • $ARG1 = src/old.ts
  • $ARG2 = src/new.ts
  • $ARGUMENTS = src/old.ts src/new.ts（完整字符串，始终可用）
3
${CLAUDE_SKILL_DIR} — Skill 自身的目录路径

用途：让 Skill 引用自己目录下的辅助文件（脚本、模板、Schema 等）。

---
description: 运行项目特定的检查脚本
allowed-tools: [BashTool]
---

请运行以下检查脚本并报告结果：

`${CLAUDE_SKILL_DIR}/check.sh $ARGUMENTS`

如果 Skill 在 .claude/skills/project-check/SKILL.md，则 ${CLAUDE_SKILL_DIR} = /absolute/path/to/.claude/skills/project-check。

4
${CLAUDE_SESSION_ID} — 当前会话 ID
 
---
description: 记录任务进度到会话专属文件
allowed-tools: [BashTool, FileWriteTool]
---

将任务状态记录到会话日志：

任务：$ARGUMENTS
日志文件：/tmp/claude-session-${CLAUDE_SESSION_ID}.log
5
Shell 注入：! 语法（正文中执行命令并内嵌输出）

在 SKILL.md 的正文（非 frontmatter）中，用反引号 !`command` 语法嵌入 Shell 命令，Claude Code 会在展示提示词给模型之前先执行这些命令并将输出内嵌进去。

---
description: 显示当前 git 状态供模型参考
allowed-tools: [BashTool]
---

当前 git 状态（自动收集）：

```!`command`
git status --short
git log --oneline -5
```

请基于以上状态，帮我生成合适的 commit message。
Shell 注入命令中可使用 ${CLAUDE_SKILL_DIR} 引用 Skill 自身的脚本文件：!`${CLAUDE_SKILL_DIR}/collect-project-info.sh`

四、allowed-tools 字段详解

设置后，Skill 执行期间这些工具无需用户确认即可使用。

allowed-tools:
  - BashTool        # Bash 命令执行
  - Read            # 文件读取
  - FileWriteTool   # 文件写入（含 Edit）
  - GlobTool        # 文件模式匹配
  - GrepTool        # 内容搜索
  - AgentTool       # 子代理
  - SkillTool       # 调用其他 Skill

允许规则（精细控制）

allowed-tools:
  - "BashTool(git *)"           # 只允许 git 命令
  - "BashTool(npm test)"        # 精确匹配
  - "Read(src/**)"             # 只允许读 src 目录
  - "FileWriteTool(*.md)"      # 只允许写 md 文件

五、context: fork — 分叉执行模式

内联模式（默认）分叉模式（fork）
任务需要看历史上下文任务自包含，不需要历史
与用户交互式后台独立完成
轻量级任务复杂、耗时的独立任务
代码审查（需要看到改了什么）生成测试（只需读源码）
fork 模式下的 token 预算：Fork 子代理有独立的 token 预算，不影响主对话的上下文窗口。这对需要读取大量文件的任务非常有价值。
---
context: fork
model: claude-opus-4-8
effort: high
---

六、when-to-use — 让模型主动调用

这是让模型无需用户输入 / 就主动调用 Skill 的关键字段。

when-to-use: |
  当用户完成了代码修改并提到要"提交"或"commit"时，自动触发此 Skill。
  当用户说"帮我提交"、"commit 一下"、"推送代码"时使用。

好的 when-to-use 写法

  • 具体描述用户的意图或措辞
  • 说明触发条件（用户说了什么、做了什么）
  • 说明不该触发的情况（避免过度触发）

七、条件激活 Skill（paths 字段）

有 paths 字段的 Skill 是条件 Skill，只有在会话中触碰（读/写/搜索）了匹配路径的文件时，才会被激活并出现在可用 Skill 列表中。

paths: "src/frontend/**/*.tsx src/frontend/**/*.jsx"

gitignore 风格的模式语法

# 匹配特定目录
paths: "src/api/**"

# 匹配多种文件（空格分隔）
paths: "*.go go.mod go.sum"

# 匹配特定文件名
paths: ".env .env.local"

# 复杂模式（换行分隔也行）
paths: |
  src/database/**/*.sql
  migrations/**
  models/**
激活后的行为：条件 Skill 激活后，在当前 session 内永久激活（不会因 /clear 重置）。user-invocable: false + paths = 完全自动触发的隐式 Skill。

八、内联 Hooks（hooks 字段）

Skill 的 frontmatter 中可以直接定义 Hooks，这些 Hooks 只在该 Skill 激活期间有效。格式与 settings.json 中完全相同。

hooks:
  PreToolUse:
    - matcher: "Bash(git commit*)"
      hooks:
        - type: command
          command: |
            git diff --cached --stat
          statusMessage: 显示待提交内容...
  PostToolUse:
    - matcher: "Bash(*)"
      hooks:
        - type: command
          command: echo "命令执行完成"
          async: true

带参数的代码审查 Skill

.claude/skills/review/SKILL.md

---
description: 对指定文件进行全面代码审查，输出改进建议
argument-hint: <file-path> [focus-area]
arguments: [filepath, focus]
when-to-use: |
  当用户要求"审查"、"review"、"检查代码"某个具体文件时使用。
allowed-tools: [Read, GlobTool, GrepTool]
version: 1.0.0
---

# 代码审查

请对以下文件进行全面审查：`$ARG1`

审查重点：$ARG2（如为空则全面审查）

## 审查维度
1. 可读性：命名、结构、注释
2. 安全性：输入验证、SQL 注入、XSS
3. 性能：瓶颈、循环优化
4. 测试覆盖：单测、边界情况

## 步骤
1. 用 Read 读取 $ARG1
2. 用 GrepTool 查找相关测试文件
3. 逐项分析，每个问题给出具体行号
4. 按优先级排序（P0/P1/P2）

调用：/review src/utils/auth.ts security

description: 为指定模块自动生成完整单元测试
argument-hint: <module-path>

effort: high
allowed-tools: [Read, GlobTool, GrepTool, FileWriteTool, BashTool]

为以下模块生成完整测试：`$ARGUMENTS`

## 流程
1. 读取源代码，识别所有导出函数/类

## 输出

5
带内联 Hooks 的 Git 提交 Skill

.claude/skills/safe-commit/SKILL.md

---
description: 带完整安全检查的 git 提交
argument-hint: "<commit message>"
allowed-tools: [BashTool]
hooks:
  PreToolUse:
    - matcher: "Bash(git commit*)"
      hooks:
        - type: command
          command: |
            STAGED=$(git diff --cached --name-only 2>/dev/null)
            if [ -z "$STAGED" ]; then echo "❌ 无暂存文件" >&2; exit 2; fi
            if git diff --cached | grep -Ei '(password|secret|token)\s*=' -q; then
              echo "⚠️ 检测到疑似机密信息" >&2
            fi
            echo "📋 待提交：$STAGED"
          statusMessage: 验证提交内容...
  PostToolUse:
    - matcher: "Bash(git commit*)"
      hooks:
        - type: command
          command: echo "✅ 提交：$(git log --oneline -1)"
          async: true
---

# 安全提交
1. `git status` 查看状态
2. `git diff --cached` 确认暂存
3. `git commit -m "$ARGUMENTS"`

如 $ARGUMENTS 为空，生成符合 Conventional Commits 的 message。

调用：/safe-commit feat: 添加用户注册功能

多文件 Skill（引用辅助脚本）

.claude/skills/security-audit/

.claude/skills/security-audit/
├── SKILL.md
├── check-deps.sh        ← 检查依赖安全性
├── check-secrets.sh     ← 检查机密泄漏
└── report-template.md  ← 报告模板
---
description: 全面项目安全审查
argument-hint: "[scope]"
context: fork
allowed-tools: [BashTool, Read, GlobTool]
---

# 安全审查

审查范围：$ARGUMENTS（为空则审查整个项目）

## 第一步：检查依赖漏洞

```!
bash ${CLAUDE_SKILL_DIR}/check-deps.sh
```

## 第二步：检查机密泄漏

```!
bash ${CLAUDE_SKILL_DIR}/check-secrets.sh
```

## 第三步：生成报告

读取 `${CLAUDE_SKILL_DIR}/report-template.md`，填入检测结果。
报告输出到 security-audit-$(date +%Y%m%d).md

check-secrets.sh 示例：

#!/bin/bash
echo "=== 机密信息检测 ==="
PATTERNS=(
  "password\s*=\s*['\"][^'\"]{8,}"
  "api_key\s*=\s*['\"][^'\"]{16,}"
  "secret\s*=\s*['\"][^'\"]{8,}"
)
for pattern in "${PATTERNS[@]}"; do
  results=$(grep -rn -E "$pattern" . 2>/dev/null | grep -v node_modules)
  if [ -n "$results" ]; then
    echo "⚠️ 发现疑似机密:"; echo "$results" | head -5
  fi
done

user-invocable: false 的隐式 Skill

.claude/skills/detect-lang/SKILL.md

不出现在 / 命令列表，但模型会根据 when-to-use 自主决定调用。

---
description: 检测用户消息中的编程语言，设置上下文
user-invocable: false
when-to-use: |
  当用户开始讨论代码时，在第一次响应前调用。

# 编程语言检测（隐式）
