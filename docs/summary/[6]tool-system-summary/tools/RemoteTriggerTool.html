RemoteTrigger 工具详解

RemoteTrigger 是 Claude Code 与 claude.ai 远程触发器（CCR, Claude Code Remote）API 对接的工具。它支持对"计划内远程 agent 触发器"的 list/get/create/update/run 五种操作。核心设计亮点：OAuth token 在进程内自动注入，永不到达 shell——这是相对 curl 的关键安全优势。每次调用（无论成功失败）都写审计记录。属于"中等复杂度"的网络型工具。

一、工具定位（一句话总结）

RemoteTrigger = 进程内调用 claude.ai CCR API 管理远程触发器的网络工具。

维度值
工具名RemoteTrigger（常量 REMOTE_TRIGGER_TOOL_NAME，prompt.ts:1）
一句话list/get/create/update/run 远程触发器，token 进程内注入 + 全程审计
是否进 system prompt❌ 受 feature flag AGENT_TRIGGERS_REMOTE 门控（tools.ts:39）
只读 / 破坏性动态——list/get 只读，create/update/run 写入（:68-70）
是否可并发✅ 可并发（isConcurrencySafe() → true，:65）
启用条件getFeatureValue('tengu_surreal_dali') && isPolicyAllowed('allow_remote_sessions')（:59-64）
核心依赖axios、OAuth（src/utils/auth.ts）、appendRemoteTriggerAuditRecord
定位互补方Bash（curl 替代品，但暴露 token）

为什么需要它？ claude.ai 提供"远程触发器"功能——可以计划在远端启动一个 Claude Code agent 执行任务。管理这些触发器需要调 API。模型本可以用 Bash + curl，但那会把 OAuth token 暴露给 shell 历史/进程列表。RemoteTrigger 在进程内注入 token，并提供统一审计，是安全合规的官方路径。

四、核心执行流程：call()

call()（RemoteTriggerTool.ts:80-172）是标准的"鉴权 → 构造请求 → 发送 → 审计"流程：

flowchart TD A["call(input, ctx)\n:80"] --> B["auditBase = {action, triggerId?}"] B --> C["checkAndRefreshOAuthTokenIfNeeded"] C --> D{"accessToken 存在?"} D -- 否 --> X["throw '请 /login'"] D -- 是 --> E["getOrganizationUUID"] E --> F{"orgUUID 存在?"} F -- 否 --> Y["throw '无法解析组织'"] F -- 是 --> G["构造 headers\nBearer token + beta header\n+ org uuid"] G --> H["switch action → method/url/data\nlist/get/create/update/run"] H --> I["axios.request\n20s timeout + abort signal\nvalidateStatus: () => true"] I --> J["appendRemoteTriggerAuditRecord\n{ok, status}"] J --> K["return {status, json, audit_id}"] I -.catch.-> L["appendRemoteTriggerAuditRecord\n{ok:false, error}"] L --> M["rethrow error"]

关键点逐条：

  1. 审计基座（:81-84）：每次调用先构造审计基座，无论后续成功失败都记录。
  2. OAuth 刷新（:86）：checkAndRefreshOAuthTokenIfNeeded() 确保 token 有效（可能自动刷新）。
  3. token 存在性检查（:87-92）：无 token 抛"请运行 /login"——友好引导而非裸错误。
  4. 组织 UUID 解析（:93-96）：getOrganizationUUID() 从 OAuth 客户端获取，作为 x-organization-uuid header。
  5. 请求头构造（:99-105）：含 Authorization: Bearer {token}、anthropic-beta: ccr-triggers-2026-01-30（:46）。token 在进程内注入，永不到达 shell。
  6. action 分派（:111-140）：list → GET、get → GET、create → POST、update → POST、run → POST /run。
  7. axios 请求（:142-150）：timeout: 20_000、validateStatus: () => true（不因 4xx/5xx 抛错）。
  8. 成功审计（:151-155）：写 appendRemoteTriggerAuditRecord，返回 auditId。
  9. 失败审计 + 重抛（:164-171）：catch 块先记录 {ok:false, error} 再 throw——审计先于重抛，保证失败也有记录。
  10. 返回（:157-163）：{status, json: jsonStringify(res.data), audit_id}。

五、权限与安全

RemoteTrigger 的安全设计是本系列的标杆：

  1. isEnabled() 双重门控（:59-64）：growthbook flag tengu_surreal_dali + policy allow_remote_sessions。前者控制功能灰度，后者控制组织级权限。
  2. 进程内 token 注入（:99-105）：OAuth token 在 Node.js 进程内组装到 header，永不经 shell。
  3. trigger_id 正则约束（:24）：/^[\w-]+$/ 防止路径注入。
  4. 全程审计（:151-171）：每次调用（成功/失败）都写审计记录，失败时审计先于重抛。
  5. validateStatus: () => true（:149）：不因 HTTP 状态码抛错——让调用方根据 status 字段决策。
  6. OAuth 自动刷新（:86）：处理 token 过期，避免手动重新登录。
  7. isReadOnly 动态判定（:68-70）：list/get 只读，create/update/run 写入，走更严格审批。

六、与其他系统/工具的关系

flowchart TD RT[RemoteTrigger] -->|HTTP| API["claude.ai /v1/code/triggers\nCCR API"] RT -->|鉴权| AUTH["src/utils/auth.ts\nOAuth token 刷新"] RT -->|鉴权| OAUTH["src/services/oauth/client.ts\ngetOrganizationUUID"] RT -->|审计| AUD["src/utils/remoteTriggerAudit.ts\nappendRemoteTriggerAuditRecord"] RT -->|门控| GB["growthbook tengu_surreal_dali"] RT -->|门控| POL["policyLimits allow_remote_sessions"] RT -.替代.-> BASH["Bash + curl\n（暴露 token，不推荐）"] RT -.配套.-> SKILL["skills/bundled/scheduleRemoteAgents\n同门控，:333"]
  • 与 CCR API：调用 claude.ai 的 /v1/code/triggers 端点，beta header ccr-triggers-2026-01-30（:46）。
  • 与 OAuth 系统：依赖 src/utils/auth.ts（token 刷新）和 src/services/oauth/client.ts（组织 UUID）。
  • 与审计系统：src/utils/remoteTriggerAudit.ts 的 appendRemoteTriggerAuditRecord 持久化每次调用记录。
  • 与 Bash：功能上可被 curl 替代，但安全上 RemoteTrigger 是官方推荐路径。

七、亮点与设计取舍

  1. 进程内 token 注入（:99-105）：本工具最核心的安全卖点。PROMPT 把它作为首要宣传点。
  2. 双重门控 isEnabled()（:59-64）：growthbook（功能灰度）+ policy（组织权限），两层独立，任一关闭即禁用。
  3. 动态 isReadOnly(input)（:68-70）：根据 action 区分只读/写入，让权限管道对 run/create 等危险操作更严格。
  4. validateStatus: () => true（:149）：不因 HTTP 状态抛错，让模型看到完整的 API 响应（含错误信息）。
  5. 审计先于重抛（:164-171）：catch 块先写审计再 throw，保证网络异常也有记录。测试专门验证此路径。
  6. trigger_id 正则防注入（:24）：/^[\w-]+$/ 简单有效，防路径穿越。
  7. shouldDefer: true：远程触发器是低频功能，延迟加载避免每个会话加载 OAuth/审计依赖。
  8. 20s 超时 + abort signal（:147-148）：网络请求标配，支持用户 ESC 中断。

九、学习建议与验证清单

  • 能解释动态 isReadOnly(input)（list/get 只读，其余写入）
