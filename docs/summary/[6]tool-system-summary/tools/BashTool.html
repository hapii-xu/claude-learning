这是工具系统逐个拆解系列中最重的一篇。Bash 是整个工具集里最复杂的工具：它有一个 47KB 的主体文件、一个 98KB 的权限引擎（bashPermissions.ts）、一个 100KB 的安全校验器（bashSecurity.ts），外加只读判定、路径校验、sed 校验、沙箱判定、命令语义、破坏性警告等十几个专项模块。这些模块合起来构成一套纵深防御的安全子系统——从 AST 解析、命令注入检测、破坏性操作告警、只读白名单到沙箱执行，层层把关。

一、工具定位（一句话总结）

Bash = 执行任意 bash 命令的执行类工具，配备完整的权限/安全/沙箱子系统。

维度值
工具名Bash（常量 BASH_TOOL_NAME，toolName.ts:2）
一句话给定一条 bash 命令，在前台或后台执行并返回合并的 stdout/stderr
是否进 system prompt✅ 在 CORE_TOOLS 白名单内（经 SHELL_TOOL_NAMES 展开，constants/tools.ts:139）
只读 / 破坏性动态判定（isReadOnly() 逐命令求值）
是否可并发⚠️ 仅当 isReadOnly() 为真才可并发（BashTool.tsx:570）
核心依赖src/utils/Shell.ts 的 exec() + src/tasks/LocalShellTask/ 后台任务系统 + src/utils/sandbox/ 沙箱
定位互补方PowerShell（Windows 对应物）、Agent（开放式探索）、Monitor（流式事件，feature-gated）

三、Tool 接口字段实现（buildTool 逐字段）

输入 schema（BashTool.tsx:301-349）

{
  command: string,                    // 必填，要执行的命令
  timeout?: number,                   // 可选，最大 getMaxTimeoutMs()
  description?: string,               // 命令的人类可读描述
  run_in_background?: boolean,        // 后台运行
  dangerouslyDisableSandbox?: boolean, // 危险地关闭沙箱
  _simulatedSedEdit?: {filePath, newContent},  // 内部字段，从模型面 schema 中省略
}

关键安全设计：_simulatedSedEdit 是内部字段，由 sed 编辑预览权限对话框设置——永远不暴露给模型（inputSchema 用 omit 剥离，:342-348）。若暴露，模型可绕过权限检查和沙箱，把任意文件写入塞进一条无害命令。

行为字段（重点）

字段实现说明
call():755核心（见第四节）
validateInput():653拦截 sleep N（N≥2）反模式，引导改用 Monitor/run_in_background
checkPermissions():666委托 bashToolHasPermission（第五节）
isReadOnly():573逐命令求值：checkReadOnlyConstraints(input, commandHasAnyCd(input.command))

四、核心执行流程：call() + runShellCommand

runShellCommand 的三种后台化触发

  1. 超时自动后台化（:1136）：命令超时时不杀进程，而是 startBackgrounding 转后台。
  2. 助手模式自动后台化（:1145-1158）：feature-gated by KAIROS。主代理中命令运行超过 ASSISTANT_BLOCKING_BUDGET_MS（15 秒）自动转后台。
  3. 用户手动后台化 Ctrl+B（:1299-1310）：进度循环里注册前台任务，用户按 Ctrl+B 时转后台。

call() 的关键点

  • sed 预览直写（:764）：若 input._simulatedSedEdit 存在，直接 applySedEdit 写文件而不执行 sed——确保预览与写入一致。
  • 退出码语义（:834）：interpretCommandResult 解释——grep 退出 1（无匹配）不算错误。
  • 大输出落盘（:882-901）：输出超过阈值时复制到 tool-results/，模型可通过 FileRead 读取。
  • Claude Code hints 剥离（:931）：<claude-code-hint /> 标签从输出中剥离，模型永远看不到——零令牌旁通道。

五、权限与安全（本工具的核心）

5.1 总入口：bashToolHasPermission（bashPermissions.ts:1641）

flowchart TD IN["input.command"] --> AST{"AST 解析\n(tree-sitter)"} AST -->|too-complex| TC["checkEarlyExitDeny\n→ 否则 ask"] AST -->|parse-unavailable| LEGACY["旧正则路径\n(bashCommandIsSafe)"] AST -->|simple| SEM["checkSemantics\n(zsh builtins/eval)"] SEM --> MODE["checkPermissionMode\n(acceptEdits 放行)"] MODE --> DENY["精确/前缀/通配 deny 规则匹配"] DENY --> SEC["bashCommandIsSafeAsync\n(注入/误解析闸门)"] SEC --> RO["checkReadOnlyConstraints\n(只读白名单)"] RO --> PATH["checkPathConstraints\n(路径/重定向/git 内部)"] PATH --> SED["checkSedConstraints\n(sed 就地编辑)"] SED --> ASK["→ ask 用户批准"]

5.2 命令注入闸门 — bashSecurity.ts

  • 控制字符（:2287）：null 字节等不可打印字符——首道闸门。
  • 命令替换（COMMAND_SUBSTITUTION_PATTERNS:16）：$()、${}、$[]、<()（进程替换）等。
  • Zsh 危险命令（ZSH_DANGEROUS_COMMANDS:46）：zmodload、emulate -c（等价 eval）等。

5.3 只读白名单 — readOnlyValidation.ts

  • UNC 路径（:1900）：Windows UNC 路径可能触发 WebDAV 攻击。
  • cd + git 复合命令（:1914）：恶意目录的假 git 钩子会执行任意代码。
  • 裸仓库攻击（:1927）：裸 git 仓库结构时运行 git 可执行恶意钩子。

5.4 退出码语义 — commandSemantics.ts

命令退出 1 含义退出 ≥2
grep / rg"No matches found"（非错误）错误
find"Some directories were inaccessible"错误
diff"Files differ"（非错误）错误

5.5 破坏性告警 — destructiveCommandWarning.ts

DESTRUCTIVE_PATTERNS（:11）用正则匹配并返回告警字符串（纯信息性，不影响权限逻辑）：git reset --hard、git push --force、rm -rf、DROP TABLE、terraform destroy 等。

七、亮点与设计取舍

  1. 纵深防御管道：权限判定不是单一检查，而是 9 层管道（AST → 语义 → 模式 → deny 规则 → 注入 → 只读 → 路径 → sed → ask）。任何一层失败都 fail-safe 到 ask。
  2. AST 优先 + 正则回退：tree-sitter 解析成功时用 AST（精确），不可用时回退正则（保守）。TREE_SITTER_BASH_SHADOW 做 shadow 测试验证新解析器。
  3. _simulatedSedEdit 内部字段隔离：从模型面 schema 中 omit，防止模型绕过权限。
  4. 三种后台化触发：超时、助手模式自动（15 秒预算）、用户手动（Ctrl+B）。
  5. 退出码语义解释：grep 退出 1 不算错误——避免模型误判"搜索失败"。
  6. 破坏性告警分层：destructiveCommandWarning 纯信息性，readOnlyValidation/pathValidation 才是硬约束。
  7. git 攻击专项防护：cd+git 复合、裸仓库、git 内部路径写入——三类钩子逃逸攻击专项检查。
  8. ReDoS 防护：MAX_SUBCOMMANDS_FOR_SECURITY_CHECK = 50 上限防复合命令指数增长冻住 REPL（CC-643）。
  9. 零令牌旁通道：<claude-code-hint /> 标签从输出剥离后记录，模型永远看不到——插件推荐等机制通过这个旁通道传递信息。

九、学习建议与验证清单

怎么读这章：先读"一、工具定位"建立心智，再跳到"四、call()"看执行编排，然后重点啃"五、权限与安全"的 5.1-5.4——这是 BashTool 的灵魂。

这是进阶篇，建议先读 GlobTool 再读本篇：GlobTool 教你"标准工具长什么样"，BashTool 教你"危险工具如何被驯服"。

验证清单（读完自测）：

  • 能说出 bashToolHasPermission 的 9 层管道顺序
  • 能解释为什么 isConcurrencySafe 直接等于 isReadOnly（只读才无副作用可并发）
  • 能指出 _simulatedSedEdit 为何从模型面 schema 中省略（防绕过权限）
  • 能说出三种后台化触发（超时、助手模式 15 秒、用户 Ctrl+B）
  • 能解释 grep 退出 1 为何不算错误（commandSemantics）
  • 能找到子命令上限（50）及其防的攻击（ReDoS / CC-643）
  • 能说出 cd+git 复合命令为何被特殊拦截（假 git 钩子执行任意代码）
  • 能区分破坏性告警（destructiveCommandWarning，纯信息）与硬约束
  • 能解释 AST 优先 + 正则回退的双路径设计
